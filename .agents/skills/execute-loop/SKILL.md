This is a shared Cursor/Codex/Claude Code Skill. Invoke it as `/execute-loop` in Cursor or Claude Code, or `$execute-loop` in Codex.

When this Skill references another shared workflow, use the current tool's sigil: `/workflow` in Cursor, `$workflow` in Codex, and `/workflow` in Claude Code.

This workflow is side-effecting. Cursor uses `disable-model-invocation: true`; Codex uses `agents/openai.yaml` with `policy.allow_implicit_invocation: false`; both require explicit user invocation. It spawns paid per-phase executors and cross-family peers and drives `/execute`, `/fix`, and commits, so it must never auto-start.

This workflow reads and writes the same `.cursor/plans/` files as Cursor and Claude Code, so no copy-paste handoff is needed when switching tools. Handoff between phases is through disk state — the plan file and git — not conversation memory.

`/execute-loop` is the capstone orchestrator: it composes the existing `/execute`, `/review-loop`, `/peer-loop`, and `/document` workflows into a per-phase build loop. It introduces NO new execute, review, fix, or document logic of its own — each phase's code is built by `/execute`, converged by `/review-loop`, cross-checked by `/peer-loop`, and documented by `/document`, with all of their gates, verification, and scope-locking intact. `/execute-loop` only decides how to spawn a fresh executor per phase, which backends run each role, when to pause for the user, and when to stop.

Its value is **context hygiene at scale**: a fresh executor per phase keeps each phase's context lean, a cross-family peer after each review catches the blind spots the executor shares with itself, and the human stays in the loop for UI/UX and every remote-affecting action. It is the reusable capstone the v27 program builds toward — the composition of `/peer-loop` (Stage 5) and `/review-loop` (v26).

## What it does / does NOT do

Runs, for each phase of an approved saved plan: spawn a fresh executor → the executor runs `/execute` on the phase's tasks, `/review-loop` to convergence in-session, and — on spawn-capable backends — `/peer-loop` in-session (a cross-family peer pass, default-on; the composer drives it as the fallback) → agent smoke → a mandatory live-user-smoke pause on UI-bearing/runnable phases → the composer runs `/document` → auto-commit (default ON; opt-out at the wizard) → discard the executor → fresh executor for the next phase, with a warn-only usage checkpoint between phases. It stops on phases-done, stalemate, cap, a MUST-PAUSE gate, a review-integrity failure, or a user stop.

Two argument-triggered submodes are the deliberate carve-outs in this otherwise side-effecting locked skill (the TOTAL dispatch in **Argument dispatch** below): exact `probe` runs the strictly READ-ONLY **accounts probe submode** — one accounts/usage table, then stop; no wizard, no run, no writes — and exact `accounts` opens the interactive account-management menu (add / delete / cancel; interactive-only, refuses headless). The lock still bars model-INITIATED invocation on every path — the submodes change what an explicitly-invoked run does, never who may invoke it.

It does NOT:
- invent execute / review / fix / document logic — the composed skills own those, with all their discipline
- bypass a MUST-PAUSE safety gate — `Defer` / `Accept` deferrals, CRIT/HIGH scope-expansion, a `/fix` stop-on-failure, the premium boundary, and the live-user-smoke pause always surface to the user; only AUTO-DISPOSABLE gate candidates (LOW/MED, non-production, do-the-work) are auto-applied and logged by the composed gate skill's headless branch (see Safety gates)
- substitute a same-family peer when the cross-family one is unavailable — it marks the peer pass inconclusive instead
- auto-push, ever — pushing stays the user's explicit call; per-phase auto-commit defaults ON, is authorized by the wizard's close-table confirm, and can be opted out at Q5
- skip the live-user-smoke pause on a UI-bearing or runnable phase
- use `cursor-agent` as a backend (its headless `-p` mode is auth-blocked — needs `CURSOR_API_KEY`)
- auto-advance across a master plan's separate stage child plans — each stage needs its own interactive Plan Mode → `/review-plan` pass first (out of scope; run `/execute-loop` per child plan)

## Roles

- **Composer** = the live session running `/execute-loop` (any tool; Cursor recommended). It orchestrates, reads phase status from the plan file (file-based handoff), never slurps full executor transcripts, and stays restartable from a run-state note in the plan.
- **Executor** = a fresh, resumable per-phase agent (Cursor subagent / `codex` / `claude -p` / the live session). It holds execution context across the phase's execute + review + peer-verify + fix — on spawn-capable backends it also drives the `/peer-loop` pass itself (loop step 4) — then is discarded at the phase boundary so the next phase starts clean.
- **Delegate (optional)** = when the executor is configured as an "architect", it spawns a same-family, pooled/cheaper-to-run premium delegate to implement spec-backed, diff-verifiable units at its own discretion, verifies each diff against the spec, and never bypasses a gate. Default OFF.
- **Peer** = a fresh cross-model-family agent per peer round (a Cursor subagent on a different family / `codex` / `claude -p`; never `cursor-agent`). Cross-family is keyed to the **executor's** family (the architect's family when delegating).

## Argument dispatch (TOTAL — resolved before the wizard)

`/execute-loop` takes one optional argument, dispatched BEFORE Step 0: the submodes inherit NO wizard step, no MUST-PAUSE, and never write `Loop config:`. The dispatch table is TOTAL:

- **No argument** → the normal setup wizard below (the dispatch itself adds nothing to this path; the v29.4 wizard items below still apply).
- **Exact `probe`** → the **accounts probe submode** (next subsection): strictly read-only — render the accounts table, then stop.
- **Exact `accounts`** → the **account management menu** (see **Account management**): interactive-only — add / delete / cancel.
- **Any OTHER argument** → print the recognized tokens (`probe`, `accounts`) and STOP — no wizard, no run, no writes: a typo must never start a run setup or a management flow.

### The accounts probe submode (`probe` — strictly read-only)

The **accounts probe submode** — named to disambiguate it from the Step 0 model probe, the preflight harness probe (step 6), and the bogus-dir isolation probe — answers "which accounts exist, on what plan, how used, and busy where?" across the default account, every eligible claude profile, and codex, as ONE table: `account | tool | plan | session % + resets | week % + resets | recent repos`. It is STRICTLY read-only: no wizard question, no run, no `Loop config:` write, no MUST-PAUSE inheritance, and no login/create/delete surface (account mutation lives ONLY behind `accounts`); the submode issues no mutation command and writes nothing itself, and account BINDING, login identity, and settings are unchanged by it — precisely scoped, not a byte-identity claim: the CLI's own internal refreshes (an enumerated incidental-write class, including an identity-preserving OAuth token rotation observed on authenticated usage probes) are disclosed in `docs/integrations/cross-agent-orchestration.md` alongside the recipes, never denied.

Rendering is truthful STAGED feedback — three fixed beats; chat surfaces are append-only, so nothing ever "fills in" an already-rendered table:

1. **The pre-call liveness line — IMMEDIATELY, BEFORE any subprocess:** composer prose naming the expected duration (the full `--usage` sweep runs ~40–70 s on a many-profile host). This line — not the roster below — carries the liveness promise on degraded hosts.
2. **The roster line:** run `python3 .cursor/bootstrap/probe-models.py` (no `--usage`) and print ONE compact NON-TABLE line from its document — accounts found (default + eligible profiles + codex presence) and the recent-repos/in-use signal. Typically fast but NOT bounded.
3. **The one table:** run `python3 .cursor/bootstrap/probe-models.py --usage` and render the final table FRESH and ATOMICALLY from that single validated document — every cell sources from this one snapshot, never from the roster call or any other account snapshot, and exactly ONE table is ever rendered.

Consume both emissions per Step 0's script contract — assert integer `probe_schema == 1` first; field names, per-cell degradations (`not exposed` for the codex account cell and absent plan tiers, `unavailable` for an unreadable recent-repos root, headline-only rows where the structured parse degraded), and the status vocabulary follow the script docstring's NORMATIVE GOLDEN EXAMPLE — the pinned consumption contract, deliberately not restated here. One renderer duty the JSON does not do for you: when `default_account.aliases_profile` names a profile row (an ambient `CLAUDE_CONFIG_DIR` aliasing an enumerated dir — the script emits BOTH rows plus the alias pointer), the pair is ONE physical store and renders as ONE row, realpath-keyed, noting both the default role and the profile identity — one physical store appears once on every account surface this skill renders (this table, the roster, and the Q1R/Q3b/Q4b/switch menus). A no-Python host gets one precise read-only line — `Python required for this submode — no probe was run; manual per-account recipes: docs/integrations/cross-agent-orchestration.md (usage probes + multi-account)` — then STOP: never a wizard entry, never a pause, no writes.

### Account management (`accounts` — interactive-only)

The SEPARATE explicit path for account mutation — never reachable from the probe submode, and never headless: a headless/loop invocation of `accounts` REFUSES LOUDLY (one line naming the interactive requirement, then stop — no menu, no writes, never a silent skip or a degraded continue). Interactive entry prints a three-option menu — `1. Add a profile`, `2. Delete (quarantine) a profile`, `3. Cancel` — each option naming its literal follow-on steps before anything runs; the menu-consent contract applies (an explicit selection here IS explicit invocation of that flow, and menus always print the literal command). The wizard's existing Q4b in-flow "new profile" guided flow stays unchanged — this menu is the standalone home, not a replacement. Full recipes live in `docs/integrations/cross-agent-orchestration.md` → multi-account (pointed at below, never restated), and `probe-models.py` stays strictly read-only — every mutation here is skill-driven interactive shell, never a script write.

- **Add (standalone branch).** The canonical binary-isolation prerequisite runs FIRST, BEFORE any target-dir creation or `auth login`: candidate enumeration + prefer-native + the bogus-dir isolation probe (Step 0's canonical pin definition, reused as a prerequisite gate ONLY — no model/plan probes and none of Step 0's pause semantics; delete and cancel stay independently reachable when it fails). Only a PASSING pinned binary reaches OAuth — when NO candidate passes, REFUSE the add with the exact remediation (install the environment-native binary — under WSL `npm install -g @anthropic-ai/claude-code` with WSL node/npm — then re-run `accounts`; the orchestration doc's readiness fix path): an account-creation operation has no safe degraded continue. Then the guided round-trip: (1) resolve the target dir — an EXISTING dir at the target path (eligible profile, quarantined store, or anything else) is a COLLISION: refuse and re-ask, never adopt or overwrite; (2) create the dir and run `CLAUDE_CONFIG_DIR=<dir> "<pinned claude>" auth login` (inline `WSLENV=CLAUDE_CONFIG_DIR/p` through a Windows shim) — the user completes the browser OAuth; (3) verify — `auth status` under the target dir reports the expected identity AND the artifact check passes: `.credentials.json` exists INSIDE the target dir (verification, never a substitute for the pre-login gate; a login that wrote to the wrong store passes `auth status` and FAILS the artifact check → REJECT the profile, surface the isolation failure, and treat the target dir as unbound). Cancel or a failed/rejected login cleans partial state — remove the target dir ONLY when this flow created it and it holds no credentials artifact; a pre-existing dir is never deleted. Default-store non-interference closes the flow: re-show the pinned binary's default-account identity and confirm it unchanged.
- **Delete (interactive-only; quarantine, never removal).** (1) Resolve the target: an EXACT enumerated NON-default profile dir via the shared eligibility predicate (the script's `is_eligible_profile_dir()` semantics — canonical realpath dirs at `~/.claude*`, excluding the built-in default store, `.lock`/control dirs, `.quarantined-` dirs, and malformed entries); the default account is never deletable — and because the predicate excludes only the BUILT-IN default store, that promise takes a second, independent check: a target whose canonical realpath equals the CURRENT default account's resolved config root (the script's env-resolved `default_account.config_dir` — an ambient `CLAUDE_CONFIG_DIR` can make the default alias an otherwise-eligible `~/.claude*` dir) is REFUSED as the live default under either path spelling; no alias of the current default store ever reaches the quarantine confirmation. A free-text path that does not resolve to an enumerated eligible dir is refused. (2) Run the BOUNDED, LIVENESS-SCOPED reference scan over three root families: (a) the current repo's `.cursor/plans/` incl. `completed/`; (b) base-side isolation journals (`.cursor/loops/*-isolation`) plus each journal's recorded authoritative worktree plan; (c) foreign repo roots discovered from ALL of the target profile's `projects/` entries (safety discovery is exhaustive — the recent subset is display freshness only), each accepted as a scan root ONLY when an AUTHORITATIVE path is recovered from evidenced entry metadata (e.g. a recorded transcript `cwd`) or independently matched to a real repo root — the lossy munged name alone NEVER resolves a root; resolved roots' plans + journals are scanned the same way, and every ambiguous/unresolved/collision token becomes a NAMED item on the INCOMPLETE branch below. Parsing rides `scan_plan_file()` semantics — heading-scoped `Loop config:` under `## Current State / Handoff Note`, fence-skipping, `## Lifecycle State` read: an operative config in an ACTIVE plan, or a journal-recorded paused/recoverable worktree authority, BLOCKS; historical configs in Completed/Archivable plans and quoted/fenced/Findings-Log text NEVER veto; comparison keys on canonical (realpath) `profile_dir` values, never display identity. (3) Add live-use evidence: a `.lock` sibling dir or a very-recent `projects/` mtime (the in-use heuristic) is live-use — refuse. (4) Dispose: ANY live reference → refuse without mutation; unreadable/undecodable roots or ambiguous ACTIVE coverage → the scan is INCOMPLETE, and deletion proceeds only through a STRONGER target-specific confirmation that NAMES the unscanned roots — never rendered "unreferenced". (5) Confirm + quarantine: resolve the quarantine destination `<dir>.quarantined-<YYYY-MM-DD-HHMMSS>` to a UNIQUE path BEFORE the final target-specific confirmation and DISPLAY it there (the exact source/destination pair confirmed is the pair renamed); the rename is NO-CLOBBER — a destination that exists or appears post-confirm REFUSES, never a silent re-target. The `.quarantined-` marker is predicate-excluded by construction, so a quarantined store vanishes from every probe/menu/`--profile-dir` surface at once. (6) Print the restore path — the exact reverse rename; restore REFUSES a collision with an existing eligible dir. No credential content is ever echoed to stdout or logs on any branch.

## Optional delegating executor ("architect")

Default **OFF**. When enabled, the executor acts as an architect: it **designs, specifies, and reviews; the delegate implements against the approved spec**. The architect holds all design and judgment work, writes a short spec for each implementation unit, offloads the implementation to a designated same-family delegate, and verifies each delegate diff before proceeding. This is still billing-pool arbitrage + context hygiene (run the metered top model less; let the delegate absorb verbose implementation output), not capability tiering — the delegate is itself premium-capable — but the offload target is **most implementation**, not just bulk-mechanical work.

Delegation heuristic: delegate a unit only if you can specify it in a few sentences and verify it by reading the diff; keep it if verifying would cost as much as writing it.

Guardrails:
- **Delegable:** any unit with an architect-approved spec that is verifiable by reading the diff — including UI implementation from the architect's design spec (layout, component structure, visual language, states, interactions) — plus bulk/repetitive edits, mechanical refactors, test scaffolding, formatting, wide find-replace.
- **NOT delegable (architect-held):** UI design; design decisions; anything touching the plan's `Critical Constraints`; security/auth; API contracts; ambiguous logic; subtle invariants where verifying ≈ re-deriving.
- The architect **verifies every delegate diff against its spec** before proceeding, and **logs misses rather than silently patching** them. The phase-smoke half of verification is backend-capability-split: a spawn-capable architect runs the smoke itself; for a headless architect the composer runs it and feeds the result into the delegation outcome log. Delegation NEVER bypasses `/review-loop` or `/peer-loop`.
- **Log every delegation.** Each decision (unit, spec one-liner, delegate backend/model) and outcome (diff verified / misses / smoke result) is a `DELEGATION:`-prefixed line in `.cursor/loops/stage-<N>-executor.log` — spawned AND live-session architects alike (a live-session architect appends directly, creating the file if absent) — and is reflected in the per-phase status line.
- Delegating mode **raises the cross-family peer pass's importance**: everything is one-family until the peer runs — do not skip `/peer-loop` when delegating.
- Keep the delegate the **same model family** as the architect, so the cross-family peer coverage (keyed to the architect) stays clean.
- Delegate **bulk, not trivial one-liners** (coordination cost outweighs the gain).
- Needs a **spawn-capable architect**. Verified spawn paths: a live session; a top-level Cursor subagent (subagents carry the full toolset including Task — nested spawn verified); or a `claude -p` session via Claude Code's **native subagent tool**, pinning the delegate `model` per spawn — children **inherit the architect's model when unpinned**, which silently burns the metered model, so pin it on EVERY delegation. Delegate-effort caveat: the native subagent tool's verified parameters are the prompt and `model` — per-spawn effort is NOT available, so delegates inherit the architect session's `--effort` (and its account profile: a native-subagent delegate runs inside the architect's session, so it bills to the architect's `CLAUDE_CONFIG_DIR` account). A shell-capable architect may alternatively delegate by invoking `claude -p` directly. `codex exec` has no in-run spawn path.

Expected outcome, honestly: big wins on UI-implementation-from-spec, boilerplate, scaffolding, and wide edits; marginal or negative on ambiguous logic; UI design/taste stays metered.

Independently of delegation, encourage the executor to **right-size effort inline** — lower reasoning effort for mechanical subtasks — as the zero-machinery cousin of delegation.

One wizard concept — attach a higher-tier **advisor** to the executor (or to the architect, when delegating): the "plan big, execute small" upgrade where a stronger model advises the advised role at its decision points while that role keeps the implementation context. The advised role is always the executor/architect — never the peer (excluded by design: peers are already one-shot cross-family reviewers). Attached at wizard Q3b, displayed on the close table's `Advisor:` line (mechanism + row + budget), persisted as the `Loop config:` `advisor=` role group. TWO mechanisms, auto-selected from the advised backend + the advisor row — the user picks WHO advises; the workflow picks HOW:

- **Mechanism N — native `--advisor` (auto-preferred when eligible).** Eligible iff the advised role is `claude -p` AND the advisor is an Anthropic model with a matrix-valid pairing AND native support is detected (acceptance-based — the Q3b lazy probe consumption and the step-6 harness assert; never help-grep); loop step 1 then renders `--advisor <model>` on the advised spawn (syntax + caveats in the Backends `claude -p` bullet). N has no separate advisor effort/profile surface — the advisor runs inside the advised session, so the stored group carries backend + model only (Q3b's mechanism-shaped item). Full-conversation quality: the advisor sees the advised session's context, not a bounded package. Capability is version-attributed: `--advisor` is accepted at 2.1.201 (observed live 2026-07-10) though HIDDEN from `--help` — detection is acceptance-based, never help-grep (the probe script's explicit `--probe-advisor` leg). **Engagement is OPPORTUNISTIC** (observed live 2026-07-10): the main model consults its advisor at its own judgment — engagement evidence (the advisor model present in the result `modelUsage`; mid-stream `server_tool_use name="advisor"` + `advisor_tool_result` blocks) proves capability, but its ABSENCE on an arbitrary prompt proves nothing, so any check that claims to verify engagement MUST use an engagement-forcing prompt (the probe script's advisor leg does; the preflight harness probe asserts acceptance only). **Pairing is enforced by the workflow, not the CLI:** the pairing matrix (the advisor must be at least as capable as the main model; haiku never advises; a fable main pairs only with fable; docs: v2.1.98+, Fable pairing v2.1.170+) is applied at the Q3b menu (matrix-filtered), asserted at Preflight step 5, and carried on the step-6 harness probe — because the CLI's two observed rejection shapes (2.1.201) make spawn-time detection unreliable: an opus←haiku pairing hard-fails (exit 1, spawn dies), but a fable←sonnet pairing is SILENTLY SOFT-STRIPPED — exit 0, a stderr-only warning, and the run continues UNADVISED, indistinguishable on stdout from a healthy unadvised run. Advised spawns therefore tee stderr (`2>&1`) so a strip warning lands in the role log (a monitor anomaly trigger string). **Failure transition (config error):** a spawn rejected — or strippable — on advisor pairing is a CONFIG ERROR, never a degradation: the workflow NEVER silently strips a close-table-confirmed advisor (the CLI would; the workflow must not) — re-ask the advisor item or user-confirm the drop, then respawn. **Billing:** N runs inside the advised session and bills ITS pool — a quality lift, not pool arbitrage (the Agent-SDK billing caveat in Preflight step 5 covers the pool; an advisored run costs materially more — observed ~2.3× an unadvised control on a trivial prompt). **N is UNCAPPED by design** — the native tool exposes no cap surface; its burn is the usage checkpoint's job (Visibility + run-state).
- **Mechanism C — the workflow consult role (every other combination).** Any advised backend × any spawnable advisor backend. A consult is ONE bounded, one-shot, foregrounded, non-interactive advisor spawn — `role=advisor`, an enum add on the existing `ROLE:` grammar (no new example shape; the spawning session writes the `start`/`end` pair per the writer rule) — that returns bounded advice TEXT to the advised role. **Transport inheritance boundary (pointer-first):** a consult inherits from loop step 4 ONLY the transport mechanics — foreground-and-wait (never background-and-end-turn), the writer + filesystem-escape checks (run by the spawning session), the truthful arming artifact on a CLI consult spawn (`WATCH: armed … role=advisor layer1=fg-wait layer2=none reason="foregrounded synchronous wait"` — the peer-spawn shape, printed by the spawning session), and the composer-driven fallback for non-spawn-capable advised backends — and EXCLUDES the peer semantics: NO cross-family requirement (an advisor MAY be same-family; HIGHER TIER is the guidance — only the peer pass requires family diversity), no `PEER_LOOP_NONINTERACTIVE` sentinel or peer-review payload, no Findings-Log writes (a consult returns advice text, never plan writes), no peer grant-list semantics (a consult spawn rides the same fixed spawn-grant shapes where the advised backend needs grants — scoped tier; moot under `perms=bypass`), no convergence/round state. **Billing:** C bills the advisor backend's OWN account/profile — the cross-pool "plan big, execute small" claim attaches to Mechanism C only. **Failure transition:** a failed or unavailable advisor WARNS-AND-CONTINUES unadvised for THAT consult — except a `[decision]` task that cannot proceed unadvised (at cap or on failure), which is MUST-PAUSE (record-and-end for a spawned executor, per Safety gates), never a silent skip.

**Consult triggers + the per-phase budget (Mechanism C).** Three trigger classes, contract-defined: a **`[decision]` task** — consult BEFORE surfacing/recording the decision, so the recommendation reaches the user advised (a task with a pre-recorded `Chosen:` line needs no consult — the decision is already made); **stuck-after-2-failed-attempts** — at `/execute`'s verification-failure boundary, consult before the hard-stop; and **ad hoc** — the advised role's judgment. ONE unambiguous per-phase budget — default **3** — counts EVERY Mechanism-C consult regardless of trigger class, failed advisor spawns included; the budget rides the spawn-prompt carry (loop step 1). The protocol is **RESERVE-THEN-INVOKE**, an append-only ledger correlated by attempt id — the ledger is the phase's `CONSULT:` lines in the phase executor's loop log (the same file as the phase's `ROLE:` lines), written by the invoking session:

  ```text
  CONSULT: reserve ts=<iso> run=<runkey> phase="<n>" id=<N> trigger=<decision|stuck|adhoc> advisor="<backend/model>"
  CONSULT: outcome id=<N> result=<ok|failed>
  CONSULT: refused ts=<iso> run=<runkey> phase="<n>" trigger=<decision|stuck|adhoc> reason=cap
  ```

  Cap-check algorithm (pinned): count THIS phase's prior `reserve` lines BEFORE reserving. Under cap → append the `reserve` line, invoke, then append the correlated `outcome` (the reserve line is NEVER mutated). At cap → append the NON-reserving `refused` line instead: no id is consumed, reservations stay ≤ cap, and refusals are counted separately — never a fourth reserve, never an uncorrelated outcome. The budget counts RESERVATIONS, so a crash between reserve and outcome stays reconstructible — a dangling reserve is a consumed attempt, classified on recovery (the Resume contract, Visibility + run-state). The advised executor's `ROLE: handoff` reports `consults=<reserved>/<cap> refusals=<n> dangling=<n>`; the composer reconciles the counters against the ledger at the phase boundary. A `[decision]` task blocked at cap is MUST-PAUSE, never a silent skip. Each phase starts at zero (fresh executor, fresh ledger). (Golden four-attempt fixture — a real consult, a failed spawn, a dangling reserve, a pre-billing refusal, and the next-phase reset: observed live 2026-07-10, run lr291-0.)

**The consult package (bounded by construction).** The advised role authors a self-contained package — the advisor has NO session context: the decision/problem statement, the minimal plan slice (the goal line + the specific task + the constraints that bind it), the options considered with their evidence, and ONE question. Bounded input — never full transcripts, whole plans, or log dumps — and bounded output: the advisor returns one advice text (a paragraph or a short list); it implements nothing, reviews nothing, and writes nothing.

## Preflight (before the first phase)

1. **Load the plan.** Confirm an approved saved plan exists in `.cursor/plans/` (not in `completed/`), read its `Lifecycle State` (must be `Active`), tasks, phases, and `Current State / Handoff Note`. If none, stop and point at `/load-plan` or Plan Mode → `/review-plan`.
2. **Cross-repo read roots (re-derived each run; no `Loop config:` key).** Scan the plan for reference paths outside this repo (a companion repo, a reference checkout the plan cites) and confirm the read-root list with the user. Carry `--add-dir <path>` for each confirmed root on every `claude -p` EXECUTOR/ARCHITECT spawn this run (loop step 1); a native-subagent delegate runs inside the architect's session and inherits its roots. Confirmed roots are READ-ONLY reference material: the spawn prompt states this explicitly (do not edit an added root unless the plan expressly scopes a cross-repo write and the user confirmed it), and the filesystem-escape baseline extends to each confirmed external git root (see Safety gates); flag a confirmed non-git root — it has no escape-check surface — for an explicit user-confirmed exception. Peers stay current-repo-only by default — a peer reviews this repo's changes; extend a peer's roots only when a phase's review genuinely needs the reference repo. When the scan finds no out-of-repo paths, this step is silent — no confirm fires. Backend limits in the Backends `claude -p` note.
2b. **Run isolation + ownership claim (a lettered sub-step, deliberately not a renumber — steps 3–11 keep their numbers; dry-run-aware).** Apply the Q5-resolved `isolation=` mode per the canonical contract in **Run isolation (worktree-per-run)** below: every editing mode performs the serialized ownership-marker claim (rescan-gated — see the canonical bullet); `isolation=branch` additionally creates or checks out its Q5-named branch in the base checkout (non-isolating — no journal, no worktree); `isolation=worktree` additionally records the base baseline, runs the pre-creation source-state gate, checks branch/path collisions, then writes the intent journal and creates the run branch + worktree in that order (journal-before-git-mutation). A dry-run PRINTS the resolved isolation actions — base/branch/path/authority — and creates NOTHING: no branch, worktree, journal, marker, plan transfer, or `Loop config:` write. Under `isolation=none` this sub-step is the marker claim only, and every other preflight step behaves exactly as it does without isolation.
3. **State the caps and scope** resolved by the wizard. Add one ADVISORY line self-assessing the composer's own context headroom — on EVERY wizard path, reuse-last and defaults included: a multi-phase run is safest orchestrated from a fresh or lightly-used session, so when this session's context is already substantially consumed, say so and recommend starting the run from a fresh composer before the first spawn. Advisory only — self-assessed (no auto-detection exists), it never blocks or gates preflight; if compaction hits mid-run anyway, the **Resume contract** (Visibility + run-state) is the recovery path.
4. **Establish the executor's model family.** When the executor's model was resolved by the wizard from probed choices — a custom menu pick, the recommended defaults, or a revalidated reuse-last config — the family is known by construction; no question needed. Only for the live-session backend, ask the user which model family is loaded — do not self-detect; the user is the source of truth. Either way, the cross-family peer family MUST differ from it.
5. **Confirm each chosen backend is installed and authenticated** (a quick version/whoami check for CLIs; model availability for subagents). Surface the **Agent-SDK billing caveat** only if a `claude -p` backend was chosen: `claude -p` currently draws from Anthropic subscription limits (the separate-credit-pool split is paused but officially announced) — verify at run time; the hedge is the Cursor-subagent backend (Cursor subscription) or a `codex` leg. When a Mechanism-N advisor is configured, also ASSERT the close-table pairing against the advisor matrix (the advisor at least as capable as the main model; haiku never advises; a fable main pairs only with fable) — load-bearing because the CLI does NOT reliably hard-stop a bad pairing at spawn: a tier-mismatched advisor can be SILENTLY stripped with exit 0 and a stderr-only warning (observed live 2026-07-10, 2.1.201), so a pairing error must be caught here and at the Q3b matrix filter, never discovered mid-run. Advisor billing extends this same caveat by pointer: Mechanism N bills the advised session's pool; Mechanism C bills the advisor's own account (see Optional advisor).
6. **Harness probe (CLI executor backends only).** Spawn a ~30-second throwaway executor on the chosen backend and have it run `git status` plus the run's exact verification commands — for `claude -p`, the verification-command grants confirmed on the wizard close table; for `codex`, the same documented gate commands they would be proposed from (no grant mechanism, but the sandbox or a hook can still block them) — both must succeed. Under `perms=bypass` the probe tests RAW command execution instead of grant shapes — the same commands, no `--allowedTools` list, with the tier's flag riding the probe spawn per loop step 1's per-invocation invariant. A bypass probe spawn BLOCKED at the spawn level (the host refuses the bypass shape itself — the spawn dies or the flag is rejected, distinct from individual verification commands failing inside a healthy spawn) routes to the ONE-pause ATOMIC scoped fallback, never a dead end: show the amended SCOPED Grants/`Permissions:` line (the shapes the scoped tier would have proposed on the close table), the user confirms ONCE, and in that same transition the resolved tier AND the persisted `Loop config:` `perms=` group normalize to `scoped` — every later spawn AND resume re-renders scoped under loop step 1's per-invocation invariant, so bypass can never silently re-enable (re-enabling it is a fresh user decision at a future wizard) — then the harness probe RE-RUNS on the actual scoped spawn shape and the run proceeds only from that proven state. A scoped probe spawn STILL blocked at the spawn level is MUST-PAUSE: `verify=composer` covers only verification-command failure AFTER a successful executor spawn — it never masks a spawn-level block. Skipped under dry-run (a dry-run spawns no executors) and for live-session / Cursor-subagent executors. On verification-command failure (inside a successful probe spawn): record composer-run verification mode up front — write the optional additive `verify=composer` run-settings key into the plan's `Loop config:` line (preflight-written, never wizard-asked; re-derived by this probe each run, so a fresh probe result supersedes any stale stored value) — and offer the wrapper-script fix (user-confirmed creation of a committed gate wrapper, e.g. `scripts/gates/typecheck.sh`; see the wrapper pattern in Backends). When a Mechanism-N advisor is configured, the probe spawn also renders `--advisor <model>` and asserts ACCEPTANCE — a clean exit with NO stderr pairing-strip warning and no hard rejection (the two observed rejection shapes, 2.1.201); it does NOT assert engagement, which is opportunistic — proving engagement requires an engagement-forcing prompt and bills a consult (the probe script's explicit `--probe-advisor` leg owns that check; see Optional advisor). An `--advisor` acceptance FAILURE here is a Mechanism-N CONFIG ERROR, not a verification-grant failure: route it to the N failure transition (re-ask the advisor item or user-confirm the drop — e.g. fall to an executable Mechanism-C row), NEVER to `verify=composer` — composer-run verification cannot repair an unsupported flag.
7. **Validate cross-family.** If the peer family does not differ from the executor family, or no cross-family peer is runnable, **mark the peer pass inconclusive and stop for direction** — never silently substitute a same-family peer.
8. **Filesystem-escape baseline.** Record the main checkout's `git status --short` so each phase can confirm headless spawns touched only expected files. Under `isolation=worktree` this baseline splits into TWO roles: the ACTIVE worktree's `git status --short` is the per-spawn edit-surface baseline (executor edits appear there), and the BASE checkout's status is recorded separately as the base-non-interference + merge-precondition proof — see **Run isolation (worktree-per-run)**; without isolation, unchanged.
9. **Tee-log rotation.** Before the first spawn, rotate any existing `.cursor/loops/` logs matching this run's key prefixes — `<runkey>-*.log`, including `<runkey>-probe.log` — with a timestamp-suffix rename (e.g. `stage-1-executor.log` → `stage-1-executor.log.<YYYYMMDD-HHMM>`), so each run's logs contain only its own lines. Tee logs append and run keys get reused (observed live 2026-07-05: a reused `stage-1-executor.log` carried a prior run's `ROLE: handoff` — a stale exit sentinel is a false anchor for the wake contract's output watch): rotation protects the Layer-1 anchor watch from stale-anchor false positives and keeps `/retro`'s per-run corpus clean. Rotation is PREFLIGHT-scoped: NEVER rotate a log a LIVE role is appending to — a mid-run rotation severs the tee and the armed watch mid-stream (observed downstream); rotation runs here, before the first spawn, and a mid-run re-rotation is legal only after the writing role is confirmed dead (the stall watchdog's check). Under `isolation=worktree` the same rotation runs INSIDE the run's per-run log root `.cursor/loops/<iso-id>/` (see **Run isolation (worktree-per-run)**). A dry-run PRINTS the intended rotations and renames NOTHING (the dry-run no-write contract).
10. **Wake-mode resolution (the `Wake:` artifact).** Resolve the host's wake predicates ONCE, here — never re-assessed ad hoc at spawn time: Layer 1 from the composer's actual spawn surface (`bg-shell` = a composer-owned background shell with an output watch; `subagent-harness` = the degraded-shell foreground-subagent harness; `fg-wait` = a composer that foreground-waits each spawn; `native` = a host-native completion surface, e.g. the Cursor running-agent card) and Layer 2 from what the host actually supports AND the run's Q5 liveness setting (`monitor` = the background monitor subagent OR a persistent harness-native watch tool — either implementation resolves as `monitor` (the two-shape clause on Layer 2, Visibility + run-state); a runnable monitor resolves `layer2=monitor` even under liveness OFF, which disables only its quiet-window returns, never its anchor/exit returns; **the Layer-2 viability check — "runnable" is PROVEN here, not assumed:** resolve `layer2=monitor` only after the IMPLEMENTATION the host will actually arm checks out on THIS composer host — the return-per-event monitor SUBAGENT needs a runnable, permitted background-subagent spawn surface and a successful lightweight trial arm; a persistent harness-native watch tool must be PERMITTED (allowlisted in the host's tool-permission surface; a tool whose calls prompt or are denied is not viable for an autonomous run — docs-only allowlist guidance in `docs/integrations/cross-agent-orchestration.md` → Host setup, never a shipped settings payload) AND its first call/arm must succeed; a gated or spawn-denied selected shape — or a failed trial/first call — makes this a NO-MONITOR host here and now — resolve `layer2=sentinel` up front (under the sentinel's own liveness-ON condition; with liveness OFF, the existing no-monitor resolution — `layer2=none` with reason) rather than discovering the dead mechanism at first spawn — and the same up-front sentinel preference applies on a host with OBSERVED monitor fragility (repeated monitor idle-outs on prior runs — the recorded-lessons signal: prefer the mechanism that works on the host over the richer one that doesn't); `sentinel` = the sleep-sentinel wake timer, selectable ONLY when Q5 liveness is ON — a sentinel IS the quiet-window wake, so a no-monitor host with liveness OFF resolves `layer2=none reason="liveness off; no monitor available"`, never a sentinel no timer will arm; `none` — REQUIRES a `reason="..."`). Print the resolved wake line in chat — REQUIRED, e.g. `Wake: run=<runkey> layer1=bg-shell layer2=monitor interval=10m` (grammar + the full artifact family in **Wake arming artifacts**, Visibility + run-state) — and write the same line as LINE 1 of each phase's `.cursor/loops/<runkey>-probe.log` when that phase's watch is first armed. A dry-run prints the line to chat only and writes NOTHING (the existing dry-run no-write contract). Loop step 1's attached-spawn mandate and the heartbeat's two-layer wake contract key on THIS resolved line for the rest of the run; unverifiable self-assessed monitoring predicates demonstrably fail (observed live 2026-07-07, run pqv29: zero probe logs written, no composer `ROLE: end` lines) — required artifacts replace them.
11. **Usage baseline (warn-only).** Probe the account pools of the CLI backends in use and report the baseline line per the usage checkpoint in Visibility + run-state (short timeout; failures degrade to "not exposed" and never block preflight).
12. **Lessons consult (appended step — steps 1–11 keep their numbers).** The composer reads `docs/lessons.md` when it exists and distills the RUN-RELEVANT items — host quirks, backend/spawn-shape gotchas, verification-routing lessons recorded there — for carry into the spawn prompts of the roles each touches (the loop step 1 carry): one line per lesson, never the pasted file, and nothing for roles a lesson does not affect. A recorded lesson a spawned role never sees cannot prevent its recurrence; `/execute`'s own one-line consult covers the executor's plan-area gotchas — this step covers RUN mechanics. No `docs/lessons.md` = silent skip.

**Setup-remediation classification (one rule for every wizard/preflight readiness or probe failure).** Generalizing the recorded reroute-don't-stall routing rule, the Step 0 environment-readiness pause shape, and the `reason=composer-run` reroute channel: classify each failure — Step 0 probes, the backend confirms (step 5), the harness probe (step 6), wake viability (step 10), the usage probes (step 11) — as **agent-fixable**, where the composer performs the fix itself (a reroute, a re-pin/re-probe, a config write), CONFIRMING with the user before any settings write, or **user-fixable**, where it pauses with the EXACT remediation steps — the file, the key, the command (the Step 0 pause's install-choice shape where an install decision exists). Config-fix scoping: per-profile ONLY when the fix lives in a profile's own config dir; a project-level `.claude/settings.json` fix is one-per-repo, never duplicated per profile. PRECEDENCE — the classifier never changes an owning probe's blocking level: it changes WHO performs a fix and how EXACT the handoff is. Probes whose owning clauses are warn-only or degradation-tolerant (the step-11/usage-checkpoint report-and-continue; Step 0's degraded free-text menus and identity-only profile displays) keep degrading and CONTINUE — exact remediation steps for them are attached as ADVISORY, never a pause; a user-fixable PAUSE fires only where the owning clause already defines a hard gate (Step 0's no-candidate-passed environment-readiness pause; a step-6 spawn-level block's MUST-PAUSE). Never a bare "preflight failed" stall: every failure resolves to a reroute, an agent fix, or an exact-steps pause — this extends the existing pause/degradation machinery above and adds NO parallel contract.

## Run isolation (worktree-per-run)

The canonical isolation contract — the lifecycle behind Q5's `isolation=worktree|branch|none` + `merge=auto|off` resolution and the close-table `Isolation:` line, executed at preflight sub-step 2b. Every isolation-conditional site elsewhere in this skill is a one-line pointer back here, and every one of them behaves exactly as before under `isolation=none` — the only machinery that runs on every editing mode (worktree AND `branch`/`none`) is the ownership marker below.

- **The isolation-run ID (`<iso-id>`).** Minted by the composer at wizard close for every editing run: collision-resistant and filename-safe (recommended shape: `<repo-or-plan-slug>-<YYYYMMDD-HHMMSS>-<short random suffix>`), and DISTINCT from the phase-scoped `<runkey>` (`stage-<N>` — the log prefix, which recurs across runs and can never safely key retained-by-default artifacts). The intent journal, the ownership marker, the run branch, the worktree path, and the per-run log root ALL derive from the iso-ID. The `ROLE:` `run=<runkey>` phase grammar is UNTOUCHED, and the journal records the run's phase-key family so recovery maps phase logs → the one isolation record.
- **Per-run log root (worktree runs).** Run logs stay under the BASE checkout, in the per-run LOG ROOT subdirectory `.cursor/loops/<iso-id>/`, holding the UNCHANGED per-phase names (`stage-<N>-<role>.log`, `<runkey>-probe.log`, `<runkey>-<role>-prompt.txt`). The journal records the log root, and rotation (preflight step 9), wake/probe paths (step 10 + Wake arming artifacts), and `/retro` ingestion all route through it — via ABSOLUTE paths carried in spawn prompts (loop step 1) — so two concurrent runs sharing a `stage-1` phase can never rotate, wake on, or ingest each other's files; the `[<runkey>]` chat breadcrumbs are per-composer-session and unchanged. Non-isolated runs keep today's flat `.cursor/loops/` paths.
- **Backend-keyed worktree location** (resolved and displayed at Q5; applied here): CLI executors (`claude -p` / `codex exec` — cwd-pinnable anywhere) → a sibling dir OUTSIDE the base checkout, default `<base-parent>/<repo>-wt/<iso-id>`; workspace-bound executors (Cursor subagent / live session — file tools confined to the workspace) → the nested gitignored `.cursor/loops/wt/<iso-id>` (legal git; the ignored container never appears in base status). A backend that cannot create or reach its keyed location makes worktree isolation UNAVAILABLE for that backend — Q5 warns and requires an explicit confirm to switch backend or drop to `none`; NEVER a silent base-checkout fallback.
- **Creation order at sub-step 2b (worktree mode; pinned).** (1) Record the base baseline — base realpath, current branch, HEAD, `git status --short` — the journal's durable fields. (2) Perform the ownership claim (below; every editing mode does this much). (3) Run the PRE-CREATION SOURCE-STATE gate: classify ALL relevant NON-IGNORED base dirt, tracked AND untracked — non-plan tracked changes, OR named untracked project inputs the plan depends on, REFUSE worktree isolation unless the user explicitly chooses a safe reversible copy/snapshot of the named files into the worktree (base copies preserved; the transfer recorded in the journal) or switches modes (`none`/`branch`, with the concurrent-edit warn) — the executor never silently builds against bare HEAD while the plan's expected source state sits uncommitted base-side; ignored runtime artifacts (`.cursor/loops/` content) are NEVER swept into a transfer set; a `merge=auto` run never starts over retained base dirt that would doom the clean-base merge precondition — normalize to `merge=off` with a note, or re-ask at close; an uncommitted/untracked ACTIVE PLAN is transferred COPY-not-move (the base copy retained; the journal records the plan-sync status) — or the spawn refuses. (4) Detect collisions: the derived run-branch name and worktree path must not already exist — a true full-iso-ID collision refuses and re-mints; existing unrelated branches/worktrees are untouched. (5) Journal-before-git-mutation: write the intent journal (state `intent`) via write-then-ATOMIC-rename, THEN create the run branch + worktree (state `created`), transfer the plan copy when required, and mark `active` before the phase-1 spawn. A failure anywhere BEFORE creation completes leaves NO partial artifacts and an unchanged base — pre-journal exits are SELF-CLEANING: on every refusal path between the claim (2) and the journal rename (5) (a source-state refusal, a collision refusal) the composer removes its OWN just-created marker before returning (the ordered-write removal rule binds only from journal creation onward), so a refusal leaves zero artifacts; a pre-journal CRASH leaves at most the marker, recovered under the journal-less worktree-marker disposition (the ownership-marker bullet below). Dry-run: print the resolved actions, create nothing (sub-step 2b's dry-run branch).
- **The registry is an ATOMIC INTENT JOURNAL** — `.cursor/loops/<iso-id>-isolation` (NO `.log` suffix: deliberately outside the step-9 rotation glob), base-side, gitignored, greppable `key=value` lines carrying discovery-critical facts ONLY — paths + authority/sync status + the base baseline + state; it never re-encodes `isolation=`/`merge=` (the `Loop config:` line owns those). State field: `intent → created → active → merging → merged → cleaned`, PLUS the durable OUTCOME states `retained-green` (the normal merge-disabled green close), `paused`, `crashed/recovery-needed`, and `merge-failed` (an ORDERLY conflicted-or-failed-merge stop — the merge was attempted, aborted, and the base restored byte/status-identical, the run branch + worktree retained; distinct from a crash, so recovery reads it as a clean retained run to re-attempt or hand off, never as `crashed/recovery-needed` and never left mid-`merging`). ORDERED writes: authority/sync/outcome are persisted BEFORE the ownership marker is removed; `merged` is persisted BEFORE any cleanup; the journal is the LAST artifact removed, and only after cleanup is VERIFIED. Recovery keys on journal facts plus reconciled git state — NEVER on marker absence or plan prose. Serialization is LOSSLESS by rule: ONE key per line; bare tokens (the iso-ID, state/mode/`plan_sync` enums, shas) unquoted; every STRING value double-quoted with backslash-escaped `\"` / `\\` / `\n` (the `ROLE:`-family quoting made explicit — absolute paths may carry spaces or `=`, and `base_status` stores the FULL `git status --short` output as ONE escaped value, `""` when clean); recovery and the merge preflight compare `base_head` and the unescaped `base_status` byte-for-byte, and `base`/`worktree`/`plan` by realpath equality — grep discovery is unchanged (every line starts `key=`). Shape (illustrative values; every transition rewrites via write-then-atomic-rename):

  ```text
  iso=<iso-id>
  state=active
  mode=worktree
  created="<iso ts>"
  base="<base realpath>"
  base_branch="main"
  base_head=<sha>
  base_status=""
  worktree="<abs worktree path>"
  branch="loop/<iso-id>"
  plan="<authoritative worktree plan path>"
  plan_sync=copied-base-retained
  log_root="<base realpath>/.cursor/loops/<iso-id>/"
  phases="stage-1..stage-<N>"
  ```

- **The ownership marker (every editing mode — worktree AND `branch`/`none`).** `.cursor/loops/<iso-id>-active`, base-side, gitignored: iso-ID, mode, checkout realpath, created-at, plus a backend-shaped liveness identity keyed to the RUN-LIFETIME OWNER — the composer/orchestrating session that holds the claim across phases (agent/session/watch identity; where the owner is itself one real dedicated process, its PID + executable/start identity) — with the current per-phase executor's PID at most an OPTIONAL, per-spawn-refreshed SECONDARY field: per-phase executors are discarded at every phase boundary, so a dead child PID is never staleness evidence against a live run. Acquisition is a serialized checkout-scoped claim whose rescan GATES activation: create THIS run's marker, then atomically rescan for other markers claiming the same checkout — a run that scans CLEAN (its own marker the only claim) proceeds unconfirmed; a run that SEES ANY contender, whenever the contender appeared, NEVER proceeds unconfirmed — it surfaces the collision for an explicit user confirm or refusal (headless: MUST-PAUSE record-and-end) — so AT MOST ONE run, the clean-scanning first claimant, ever becomes an unconfirmed editor. The deterministic iso-ID tie-break (lowest byte-order) ORDERS the surfaced arbitration only — it names the presumptive owner presented at the confirm — and never licenses a contender-observing run to proceed on its own. Stale-marker reconciliation is CONSERVATIVE: unknown liveness ⇒ warn/refuse, never delete-as-stale by guess. The marker is removed on close — for worktree runs only AFTER the journal's authority/sync/outcome write (the ordered-write rule); on crash it is reconciled, never blindly deleted. A journal-less marker recording `mode=worktree` marks the PRE-JOURNAL crash window (the claim-to-journal-rename gap at sub-step 2b — no journal, run branch, or worktree exists yet) and its disposition is TOTAL: validate the recorded run-lifetime owner exactly as above — live or unknown liveness ⇒ retain + warn/refuse; a conclusively-dead owner with ZERO matching git/isolation artifacts (no journal, no run branch, no worktree at the iso-ID's derived names) ⇒ remove ONLY the ownership-proven marker, base untouched; ANY matching artifact found ⇒ retain + surface (the journal contract owns recovery from there). The `branch`/`none` concurrent-edit warn (Q5) keys on THESE markers — never on broad `pgrep` sweeps or stale log prefixes.
- **Authority + the per-close-path outcomes (stated ONCE — here).** The WORKTREE plan copy is AUTHORITATIVE from spawn: run-state + `Loop config:` writes target it from creation onward, composer git duties and fallbacks (peer, verification, smoke, `/document`) run against the worktree, and the escape baseline splits per preflight step 8. Close paths — every termination resolves through exactly one row:
  - **Verified merge-back** (green close — the Phases-done termination — with `merge=auto`) → the BASE becomes authoritative again; cleanup may proceed (worktree removed, run branch deleted, journal → `cleaned` only after verification).
  - **Green close, merge disabled** (`merge=off`) → RETAIN worktree + branch + journal (state `retained-green`); the worktree plan stays authoritative; the handoff states the continuation (how a later session resumes or merges).
  - **Pause / crash / resume** → worktree authority is KEPT under the journal contract (`paused` / `crashed/recovery-needed`); any resume reconciles the journal against `git worktree list`, refs, the recorded base state, and any extant close lock (the close-lock recovery rule below) BEFORE any merge, authority switch, or deletion; ambiguity retains ALL artifacts and stops without touching the base.
  - **Pre-creation failure** → NO partial artifacts; unchanged base.
  - **Any non-green close** → the worktree + branch are RETAINED, recoverable — never auto-cleaned.
- **Merge-back preconditions (`merge=auto`; composer-run; LOCAL only).** Merge the run branch into the base branch ONLY under the BASE-SCOPED CLOSE LOCK and ONLY against a CLEAN, UNMOVED base. The close lock — `.cursor/loops/close.lock` in the base checkout, taken by atomic create with write-then-atomic-rename content naming the holder's iso-ID + created-at — is acquired FIRST, and the clean/unmoved predicate is RE-READ under it: base status clean AND base HEAD still at the journal's recorded baseline, verified POST-acquisition (any pre-lock check is advisory — the predicate and the merge must be inseparable, or two green runs can both pass at the same HEAD and the second merges onto a moved base). The lock is held through the journal's `merging` write, the local merge, verification, the `merged` + authority persistence, and the cleanup decision, then released. A CONTENDED or ambiguously-stale close lock RETAINS the run's artifacts and STOPS without touching the base — stale-lock reconciliation is CONSERVATIVE, keyed on the named holder's journal/marker, never delete-by-guess (the marker rule). A post-lock predicate failure (moved or dirty base) likewise retains the run branch and STOPS — never an opportunistic merge onto the new HEAD. CLOSE-LOCK RECOVERY (a fresh session finding an extant `close.lock`; the disposition is TOTAL): validate the recorded holder FIRST via its named iso-ID's journal + ownership marker — (a) the holder is THIS resuming run (same iso-ID, live owner) → ADOPT the own lock and continue from the journal's state; (b) the holder is PROVABLY DEAD (its run-lifetime owner conclusively gone) → reconcile under an exclusive recovery acquisition against journal + base + run-branch state: a pre-merge journal state with the base unchanged → RELEASE the lock safely (a later close reacquires); journal `merging` with the merge NOT STARTED — base HEAD/status still byte-matching the recorded clean baseline AND no git merge metadata/index residue present → classify NOT-STARTED and either release safely (a later close reacquires and re-runs the post-lock predicate) or continue under the exclusive recovery ownership after re-running that predicate; journal `merging` with the merge PROVABLY COMPLETE (base HEAD is the run branch's merge result and verification passes) → persist `merged` + base authority, then release; journal `merged` with only the release pending → complete the release (idempotent convergence); anything ambiguous, dirty, or genuinely mid-merge → RETAIN + STOP without touching the base; (c) unknown liveness → the conservative rule: warn/refuse, never delete. Lock removal is always OWNERSHIP-CHECKED — only the recorded holder or the exclusive recovery owner removes it, never age-based. A failed or conflicted merge is ABORTED and the base restored byte/status-identical — the base is never left mid-merge — and the journal records the `merge-failed` outcome state (the orderly-stop enum value: never left at `merging`, never mislabeled `crashed/recovery-needed`), the run branch + worktree retained for a re-attempt or handoff. Worktree, branch, and journal are retained until merge VERIFICATION passes (retention-until-verified); cleanup is IDEMPOTENT (re-running a partial cleanup converges, never errors); and merge-back NEVER pushes — push stays the user's explicit call on every path.
- **`branch` and `none` are NON-ISOLATING.** `branch` = the `none` path plus a named branch — created or checked out in the base checkout at sub-step 2b — the run edits the SHARED base checkout, and there is NO journal, worktree, or merge-back lifecycle. Both modes still write the ownership marker, and when another editing run's marker claims this checkout, Q5 warns + requires an explicit confirm — refusal recommended. Fresh-session recovery for isolated runs starts at the Resume contract's registry-discovery step 0 (Visibility + run-state); the fresh-session entry skills carry their own registry-awareness clauses.

## The per-phase loop

Each phase is: spawn a fresh executor → build → converge → cross-check → smoke → pause-if-needed → document → commit (default ON) → discard. Start at phase 1.

1. **Spawn a fresh executor** for this phase on the chosen backend (a subagent prompt points at `.agents/skills/execute/SKILL.md`), using the wizard's resolved configuration: the chosen model, the effort level only if one was explicitly picked (inherit = no effort flag), and — for a `claude -p` backend — the `CLAUDE_CONFIG_DIR=<profile dir>` prefix (the invocable path — the `Loop config:` line's `profile_dir`, not the display identity). Two spawn invariants on every CLI spawn: (i) the working directory is PINNED EXPLICITLY — the invocation sets its cwd to the intended checkout (the repo root; the journal-recorded worktree under isolation), never inheriting the composer's ambient shell cwd (an inherited cwd points the role at the wrong tree the moment composer shell state drifts — observed live downstream); toolchain-path pinning for version-managed node on WSL hosts rides the same spawn-shape discipline (canonical guidance: `docs/integrations/cross-agent-orchestration.md` → Host setup); (ii) ONE live executor per phase — before spawning, confirm no prior spawn for this phase is still live (its shell/PID confirmed dead per the stall watchdog's check, never assumed from silence): a double-spawned phase has two roles editing and logging over each other. Under `isolation=worktree`, every executor/architect (and delegate) spawn is cwd-pinned to the journal-recorded WORKTREE path, and the spawn prompt carries the run's ABSOLUTE base-side log paths — tee, probe, and prompt files live under the base checkout's per-run log root, never the worktree; tee redirection stays composer-owned base-side (see **Run isolation (worktree-per-run)**). A `claude -p` spawn invocation also passes the run's grants as `--allowedTools` flags — the fixed peer-spawn grant list (loop step 4) PLUS the verification-command grants confirmed on the wizard close table, and never more: a grant the user has not seen and confirmed on the close table is never passed (the wizard's injection gate) — and adding ANY grant after the close-table confirm (at a later phase's spawn, a respawn, or mid-run) requires a fresh user confirm of the amended grant list before it is passed (observed live: a `Bash(cp *)` grant slipped into a phase-2 spawn unconfirmed). The grant machinery is the SCOPED tier (`perms=scoped`, the default). Under `perms=bypass` the spawn renders the resolved tier's flag INSTEAD of any `--allowedTools` list — `claude -p`: `--dangerously-skip-permissions`; `codex`: `--dangerously-bypass-approvals-and-sandbox` (see Backends) — and **permission rendering is a PER-INVOCATION invariant**: every fresh spawn, respawn, AND resume of the executor/architect renders the resolved tier's flags — resume sites included: composer-run verification resumes (step 3), smoke-feedback relays (step 6), Mechanism-C advice returns (Optional advisor), and usage-recovery resumes/account transfers — because the tier never survives implicitly from the prior process (observed live 2026-07-10: a resumed `claude -p` session without the re-passed flag came up `permissionMode: auto`, and codex's resume command carries its own bypass flag) — all WITHOUT re-asking the already-confirmed tier; genuinely NEW grants keep the fresh-confirm rule above. When preflight confirmed cross-repo read roots (Preflight step 2), the spawn invocation also carries them as `--add-dir <path>` flags, and the spawn prompt states they are read-only reference roots (Preflight step 2's guardrail). The composer writes the executor's `ROLE: start` line when spawning (and the `ROLE: end` line on completion), per Visibility + run-state. When architect mode is ON, the spawn prompt must additionally carry the delegate backend + model and point at this skill's `Optional delegating executor` section for the delegation rails + logging contract — subagents do not auto-load slash skills (see Backends), so without this a spawned architect never sees the rails. A live-session executor skips the spawn (and a live-session architect reads the section directly — no prompt carry needed). When an advisor is configured (`Loop config:` `advisor=`): a Mechanism-N spawn renders `--advisor <model>` on the advised invocation; a Mechanism-C spawn prompt carries the advisor row (backend / model / effort / profile_dir), the consult triggers, the bounded package spec, and the per-phase budget with the RESERVE-THEN-INVOKE `CONSULT:` ledger obligation, plus the exact section pointer (**Optional advisor (any backend)**) — subagents do not auto-load skills, so without this carry the wizard's advisor selection never reaches the executor; a live-session advised role reads the section directly. A non-spawn-capable advised backend (`codex exec`) or a failed in-executor consult gets the composer-driven fallback, with the advice returned over the pinned per-backend route (the transport matrix). The spawn prompt also carries what the in-executor step-4 peer pass needs, so it can run without asking a user: the `/peer-loop` skill path (`.agents/skills/peer-loop/SKILL.md`), the executor's own model family (known by construction from the wizard), the resolved peer row (backend / model / effort / profile_dir), the peer round cap, the run key, the wake-artifact obligation for its own peer spawns — an exact pointer to **Wake arming artifacts** (Visibility + run-state) plus the instruction that each in-executor foregrounded peer spawn prints its OWN truthful armed line, `WATCH: armed … role=peer layer1=fg-wait layer2=none reason="foregrounded synchronous wait"` (fg-wait by construction per loop step 4; never a copy of the composer's outer `Wake:` state) — and, for a `claude -p` executor, the fixed peer-spawn grant list (see loop step 4). For a SCOPED `claude -p` executor the spawn prompt additionally carries the close-table-confirmed verification-command grant shapes VERBATIM, with the instruction to run verification ONLY in those exact shapes (or via the committed wrapper-script path when one exists): a variant of a granted command — an added env-var prefix, a re-ordered flag — is denied on every attempt (observed live: a `TMPDIR=/tmp`-prefixed variant of a granted test command was denied on every re-verify in a downstream run, with the grants sitting unused). Under `perms=bypass` that carry is replaced by one line — run verification directly; no grant-shape restriction applies. It also states the two exit rules a headless executor cannot infer: EVERY child it spawns — peer rounds first among them — runs foregrounded-and-waited (never background-and-end-turn — see loop step 4), and the run's final output line must be the `ROLE: handoff` exit sentinel (see Visibility + run-state). The spawn prompt also carries the run-relevant lessons distilled at Preflight step 12 — one line per lesson, only the roles each touches — so a recorded gotcha reaches the role that could repeat it. The spawn prompt also carries the current wall-clock time at spawn — a headless session does not know wall clock reliably, so this is the role's timestamp anchor (spawn time + elapsed) for its self-written `ROLE: handoff` `ts=`, which stays advisory per the timestamp-authority rule in Visibility + run-state. **Attached spawn (a host-conditional mandate keyed to the preflight-resolved `Wake:` line — Preflight step 10; the host predicate is resolved THERE, never re-assessed per spawn).** Under `layer1=bg-shell` (e.g. a Cursor composer session), a CLI executor/architect MUST run as a composer-owned background shell — teed to its role log, never `nohup`-detached or otherwise disowned — with the output watch configured AT SPAWN on the full Tier-1 anchor set, and the resolved Layer-2 mechanism armed at the same moment, per the two-layer wake contract in Visibility + run-state: the shell's completion notification is the exit wake (in a repo whose content documents the anchor vocabulary, anchor matches are advisory — key exit on shell completion + the role's own final line, per the wake contract's meta-repo caveat), and **a spawn wrapper exiting 0 is not a watched executor** (observed live 2026-07-05: a wrapper detached the PID, the composer owned no watchable shell, and the executor's `ROLE: handoff` sat unread for hours). A headless or non-persistent composer has no such surface (`layer1=fg-wait`) — it foreground-waits the spawn, with the sleep-sentinel wake timer covering quiet windows where resolved (`layer2=sentinel`). **Harness-spawn variant (degraded composer shell — `layer1=subagent-harness`):** when the composer's own shell surface is degraded but subagent shells work (the Safety-gates degraded mode), spawning the CLI role inside a FOREGROUND shell subagent that waits on it preserves Layer 1 — the subagent's completion notification becomes the exit wake — with the Layer-2 monitor unchanged and the same exact-commands-only harness contract (verified live 2026-07-06: a pool-reset resume and a full phase both ran end-to-end through foreground subagent harnesses after the composer shell died). The one legitimate detached exception — a spawn that must survive composer restart — makes the Layer-2 monitor REQUIRED for that spawn, not optional. **Arming artifact (REQUIRED on every CLI-role spawn).** The composer MUST print one `WATCH: armed` line at the moment the spawn's watch is armed — e.g. `WATCH: armed ts=<iso> run=<runkey> role=executor layer1=bg-shell layer2=monitor` (grammar in **Wake arming artifacts**, Visibility + run-state) — carrying the MANDATORY `layer1=`/`layer2=` state fields — the ACTUALLY-armed state, which normally matches the resolved `Wake:` line; the phase's FIRST armed line also writes the `Wake:` header as line 1 of that phase's `<runkey>-probe.log` (Preflight step 10). When the resolved Layer-2 mechanism fails to arm (e.g. a monitor spawn error), fall back to the sentinel and print the armed line as `layer2=sentinel reason="..."` naming the failed monitor arm — the actual mechanism, never the resolved-but-dead one; when the sentinel fails too, the line still prints — as `layer2=none reason="..."` naming the failure and stating that quiet-hang detection is OFF — so degradation is loud by construction: there is no separate degraded verb, a spawn with no printed arming artifact is an unwatched spawn (a contract violation, never a silent fallback), and the `layer2=none` continuation is legal only under an actual Layer-1 surface per the return-handling duty in **Wake arming artifacts**. After spawning a CLI executor (or architect), **surface its live streaming terminal once** so the user can watch the build breathe — on a Cursor host, via the `cursor-app-control` `open_resource` tool as a **fallback chain, never failing silently**: target the role's tee TERMINAL resource first; if that fails, try the tee-log file URI; if both fail, print the tee-log path in chat as the last resort (observed live: the file URI was rejected with "file URI must be inside the current workspace" even for an in-workspace path, and the run went dark). An orchestrator without such a surface prints the tee-log path instead. Surface at spawn only — never re-focus mid-run (repeated focus-stealing is worse than silence). The Cursor-subagent backend needs nothing here: its running-agent card is the native working indicator. Composer-driven fallback peer spawns (loop step 4) also need nothing — they are short foregrounded rounds, already visible as a running shell call.
2. **Build the phase** — the executor runs `/execute` on this phase's tasks only, honouring every `/execute` hard-stop (decision-point tasks, premium boundary, verification failure, safety boundary). If the architect mode is on, the executor may delegate spec-backed, diff-verifiable units per the guardrails above.
3. **Converge** — the **executor** runs `/review-loop` in-session to convergence on the phase's changes (its own gates and cap apply): it holds the phase's execution context, and the composer only orchestrates — it never runs the review itself. Independence comes from the step-4 cross-family `/peer-loop`, not from moving the review to the composer. **Verification runs before review convergence:** when the phase's verification suites are composer-owned (an ungranted backend or a `verify=composer` run — see step 5), the executor ends its run with `ROLE: handoff reason=composer-run suites="..."` BEFORE starting `/review-loop`; the composer runs the named suites and resumes the executor (backend-correct resume syntax — claude `--resume <session-id>`, codex `codex exec resume <session-id>`) with the results — review convergence over tests that were written but never executed is not convergence (observed live: a review + peer round converged on never-run DB suites). Granting the pure non-DB/non-network suites via scoped `--allowedTools` (see Backends) shrinks these round-trips to the composer-owned suites only. Carve-out: on a genuinely docs-only phase the composer MAY run the review directly — with caution: in the v27 Stage 8 C2 live run, a composer diff-review found 0 findings where the in-executor review caught a real LOW; any phase touching code stays in-executor. At step-3 completion the composer asserts that `/review-loop` reported its per-round `Review History` integrity check (see `/review-loop` loop step 1); if the report is missing, stop the loop and surface it.
4. **Cross-check** — run `/peer-loop` (default-on per the wizard cadence): a fresh cross-family peer runs `/peer-review` headless → each finding is verified against the code and `/fix`ed → a fresh peer re-reviews → to convergence/stalemate/cap. Skippable only for genuinely trivial phases. **Who drives it is capability-conditional:**
   - **Spawn-capable executor (primary): the executor runs `/peer-loop` in-session**, accepting `/review-loop`'s convergence chain-offer directly per the wizard cadence instead of offering it to an absent user — step 3→4 is one continuous in-executor flow, and the executor (which holds the phase context) verifies findings and applies `/fix` without per-round composer round-trips. This is `/peer-loop`'s primary topology (its composer = the executor); the executor supplies `/peer-loop`'s orchestrated-mode config from the spawn-prompt carry (step 1): its own family, the resolved peer row, the cap, and the run key. Spawn-capable backends: a Cursor subagent (Task tool), the live session, and — verified live 2026-07-03 — a `claude -p` executor granted the **fixed peer-spawn grant list**: `Bash(codex exec:*)` + `Bash(git status:*)` + `Bash(git diff:*)`. The grant list is fixed per backend pair and NEVER derived from plan text (agent-authored plan prose is an injection vector for `--allowedTools`); it is displayed in the wizard close table and logged in the executor's `ROLE: start` context. The grant list is a SCOPED-tier concept — under `perms=bypass` it is MOOT: a bypass `claude -p` executor is always peer-spawn-capable, no grant needed (and the ungranted-executor fallback below does not arise).
   - **Non-spawn-capable executor (fallback): the composer drives `/peer-loop`** exactly as before, routing each finding to the executor to verify and `/fix` — under `isolation=worktree`, against the WORKTREE checkout: the diff and the authoritative plan live there (see **Run isolation (worktree-per-run)**). Applies to `codex exec` (its sandbox denies network for a nested spawn — a scoped-tier rationale; whether the no-sandbox bypass form restores a nested spawn path is UNVERIFIED, so the composer-driven fallback stands for codex on BOTH tiers until live-probed), to an ungranted SCOPED `claude -p` executor (a moot case under bypass — see above), and to a `claude -p` executor running degraded after Step 0's environment-readiness pause — its binary failed the isolation probe, so the pause's no-in-executor-peer-pass consequence stands regardless of grants; the composer drives the pass.
   - **Grant-failure fallback:** if the executor's peer spawn fails (grant denied, CLI missing, spawn error), the executor reports the failure and the composer drives `/peer-loop` instead — the peer pass is never skipped and never marked converged on a failed spawn.
   - **Peer spawns run foregrounded — never background-and-end-turn.** A headless executor terminates at turn end: a peer round spawned in the background and "awaited" via a completion notification is orphaned the moment the executor ends its turn (observed live, 2026-07-03 — the run died mid-pass; RECURRED 2026-07-19 downstream despite the rule, so it is stated mechanically): each peer `codex exec` runs in the foreground of its shell call and is waited on synchronously — no `&`/`nohup`/disown, no harness background flag, no spawn-then-check-later — and the same rule binds EVERY child a headless role spawns (peer rounds, Mechanism-C consults, a shell-invoked delegate), not peers alone. When one foreground wait cannot span a long child (a harness per-call time cap), stay in the SAME turn and re-enter bounded foreground waits — each poll command bounded ≤2 minutes (the proven headless-poll ceiling: a longer single wait trips harness caps and reads as a hang to watchdogs) — until the child completes; ending the turn while the child runs is the failure, whatever the intent to check back.
   - **Composer takeover mid-pass.** If the executor dies mid-`/peer-loop` (crash, limit-death, an orphaned round), the composer resumes `/peer-loop` from the plan's Review Findings Log state: round numbering continues from the last logged round, and the composer closes any open round block, branching on the round's `Source:` — a code peer round gets today's behavior (verify outstanding findings against the code, drive `/fix`); a plan peer-review round (`Source:` ending `plan peer-review`) resumes the planning-session amendment/disposition path with plan-diff convergence instead — never `/fix`, never code verification. Never restart the pass from round 1 and never mark it converged on an interrupted round.
   - **Escape checks follow the spawning session:** whoever spawns the peer inherits `/peer-loop`'s per-round `git status` filesystem-escape checks — an in-executor peer pass means the executor runs them.
   - **MUST-PAUSE inside an in-executor peer pass:** the spawned executor records the open gate (plan handoff note + its final message) and ENDS its run; the composer relays it to the user — a spawned executor never blocks in-session waiting for a human (see Safety gates).
5. **Smoke** — run the phase's agent smoke (tests / build / the payload-sync check where applicable) per the plan's `Validation / Verification`. When the executor backend cannot run approval-gated shell commands (and has not been granted them via scoped `--allowedTools` — see Backends), or the run carries `verify=composer` (the preflight harness probe proved the granted commands do not work — Preflight step 6), the composer runs the smoke. Two routing/advisory rules from live runs: suites that source `.env` (DB/integration suites reading secrets or connection strings from the environment) are COMPOSER-RUN by default regardless of grants — secrets never ride spawn grants, an env-prefixed invocation breaks exact-match grant shapes (see Backends), and a sandboxed role cannot reach localhost services anyway; and when a runnable smoke needs a dev server in a multi-app repo, CHECK THE PORT first — an in-use port means a live instance already exists (the user's own, another run's): reuse it or surface the collision, never start a fighting second instance or blind-kill the holder. Under `isolation=worktree`, composer-run smoke — like every composer-run verification suite this run, the step-3 `reason=composer-run` suites included — executes against the WORKTREE checkout (see **Run isolation (worktree-per-run)**).
6. **Pause if needed** — if the phase touched UI-bearing or runnable behavior, OR any MUST-PAUSE gate fired (see Safety gates), **hard-pause and surface to the user** for a live user smoke test and/or the gate disposition. Do not proceed on the user's behalf. Expect UI-bearing phases to be **multi-cycle/assisted**: "look → tweak → look again" is the normal shape; a single pause is the floor, not the model. **Smoke-pause routing default (message interpretation while the pause is open).** The composer's DEFAULT interpretation of ANY user message during this pause is smoke feedback — batched and relayed to the executor for assessment WITHOUT the user having to say so ("send this to the executor" is implicit, never required wording). PRECEDENCE runs before the default, in order: (1) a clear answer to a currently-OPEN composer-surfaced prompt — a gate disposition ("1", "accept all recommendations", "override item 3"), a usage-recovery choice, a user-only environment-verification answer — is consumed by that open prompt and never relayed as feedback; a MIXED message ("2; also the Save button is still clipped") consumes the disposition for the open gate and relays the remaining feedback verbatim; (2) an explicit run-mechanics directive addressed to the composer (pause/stop/cap/config changes, a direct question about run state) is handled by the composer. Only after both checks does the feedback default apply — and only then does genuine ambiguity RELAY-WIN. The composer acknowledges with ONE line naming the batched items and the relay — never "want me to relay this?" (the duties-are-never-asks style, Safety gates) — and never assesses, answers, or acts on the feedback itself. **Clarification ownership:** the composer never originates content-clarifying questions about feedback — it batches the user's wording VERBATIM, marking unresolved uncertainty for the executor to assess; when the EXECUTOR raises a clarification, the composer relays that executor-authored question to the user and routes the answer back without interpretation (a headless executor raises it via the record-and-end path — the Headless seam below); run-mechanics questions stay composer-handled. **Smoke-feedback triage (routing contract).** When the live-user-smoke pause returns feedback, never apply it ad hoc — route every item, however small it looks: the composer batches ALL feedback items and relays the batch to the phase's executor (backend-correct resume syntax: `--resume <session-id>` is claude-only, a codex executor resumes via `codex exec resume <session-id>`; a live-session executor receives it in-session; an executor that cannot be re-engaged is replaced by a fresh executor spawned from the plan handoff note — plan state is on disk, so no context is lost). **The composer NEVER implements feedback fixes itself — relay is the whole duty** (user-reported 2026-07-10: composers patched smoke feedback directly; positive routing alone did not hold). Recorded rationale, one sentence: a composer-authored fix bypasses the phase's review + peer gates, guesses without the executor's build context, drains the orchestration session's context on implementation output, and — on a deliberate billing split — burns the composer's pool for the executor's work. Role-binding, not exceptions: a live-session executor IS the executor role (the same session implementing is that role doing its job), and a user-confirmed usage-exhaustion takeover (option 3 in Usage-exhaustion recovery) REASSIGNS the executor role to the live session — both are role reassignments, not prohibition exceptions; `verify=composer` and the degraded-shell exact-commands mode stay out of scope — running named verification commands is not authoring fixes. The executor runs the batch through `/execute`'s Deferral Confirmation Gate, presented — for user-authored smoke feedback ONLY — as **one compact table**: every item with a recommended disposition (`Include in plan` for anything UI-touching or potentially multi-surface; `Fix now` — executor-implemented like every accepted disposition here — reserved for genuinely trivial single-site items, still design-doc-bound when UI-touching), then a single confirm with per-item overrides ("accept all recommendations, or override by number"). The batch table is an explicitly-scoped presentation variant: these items are the user's own requests given moments earlier, not agent-discovered risks, so one-at-a-time prompting is fatigue without safety value — the canonical one-at-a-time gate shape is unchanged everywhere else. `Include in plan` items append 🟥 task(s) to the plan, each carrying a user-smoke provenance note (which pause, verbatim ask), and the whole batch then gets ONE scoped `/review-plan` per `/execute`'s `Include in plan` handler and `/review-plan`'s **Scoped mode** (the canonical definitions — not restated here); that scoped pass consults the design doc for UI-touching task(s) and enumerates sibling surfaces app-wide (feedback names an INSTANCE of a pattern; the Coverage lens finds every surface the pattern touches — the observed failure mode is making the named column sortable and missing the rest). The executor then implements via `/execute` and the user re-smokes — the multi-cycle shape above. Headless seam: the gate items follow the headless auto-disposition contract, and a clarification the executor cannot answer records-and-ends per MUST-PAUSE surfacing (both in Safety gates).
7. **Document** — the **composer** runs `/document` (interactively, so its confirm gates — lessons distillation, the bloat guard, plan sync — work), explicitly naming the governing plan. This is deliberately composer-run, not executor-run: the executor's plan/handoff updates have already landed on disk by this step, and `/document` is git-diff-driven, so the composer loses no needed context. Under `isolation=worktree` the composer runs it against the WORKTREE checkout — the phase's code and the authoritative plan copy live there (see **Run isolation (worktree-per-run)**). The run-end interactive `/document` (see Termination) owns the final lessons + bloat-guard pass.
8. **Commit (default ON)** — unless auto-commit was opted out at the wizard, commit only this phase's files per **Auto-commit**; if opted out, leave the changes staged-for-review and note it. Never push.
9. **Discard + advance** — discard the executor. If more phases remain AND scope allows auto-advance AND no stop condition fired: run the usage checkpoint (probe the CLI pools in use and report the warn-only line — see Visibility + run-state; never pause on the numbers), update the run-state note (see Visibility + run-state), then return to step 1 with a fresh executor for the next phase.

## Phase detection + bounded discretion

- **Unit = a phase** (a plan-defined task group). If the plan groups tasks into phases, use those. If the plan is a flat task list, the composer groups **contiguous low-risk tasks** into a phase and **isolates** foundational, security, schema, or shared-invariant tasks into their own early phase.
- **Plan-time grouping decides where reviews happen** — the premium planner made that risk judgment with full context, so it is auditable and deterministic.
- **Run-time discretion is add-only:** on a surprise (a phase turns out riskier than planned), the composer MAY checkpoint EARLIER — split a phase, review sooner — but MUST NEVER skip a planned checkpoint or merge a foundational task into a batch to avoid a review.

## Safety gates — the loop PAUSES, it never bypasses

Every gate of the composed workflows is inherited and honoured. Under a headless/loop invocation the loop pauses for every MUST-PAUSE case and never auto-resolves it; only AUTO-DISPOSABLE candidates are auto-applied and logged, per the three-way contract below.

- **MUST-PAUSE (headless auto-disposition contract).** Under a headless/loop invocation, only AUTO-DISPOSABLE gate candidates flow automatically: direction is do-the-work (`Fix-now` / `Include in plan`) AND severity is LOW/MED AND not production-impacting — the executor applies its own recommendation and **logs** it. Everything else is MUST-PAUSE and surfaces to the user, never auto-decided: `Defer` / `Accept`; CRIT / HIGH; production-impacting (new dependency / env var / migration / deploy-config / parity drift); a `/fix` stop-on-failure; or genuine uncertainty. (This is the same three-way classification the composed gate skills carry.)
- **Live-user-smoke pause (mandatory).** On any UI-bearing or runnable phase, hard-pause for a live user smoke test before document + commit. This is the autonomy ceiling for UX. Feedback from this pause is never applied ad hoc and never composer-implemented — it routes through the smoke-feedback triage contract, its routing default, and the composer prohibition in loop step 6.
- **Premium boundary.** A `[executor: premium-only]` task inside a phase hard-stops per `/execute`; confirm the executor backend is on the intended tier before continuing.
- **Never auto-push.** Pushing is always the user's explicit call.
- **Filesystem-escape check.** Record the main checkout `git status` before and after every headless spawn; a peer should touch only the plan file, an executor only the phase's files. When the run carries confirmed cross-repo read roots (Preflight step 2), the baseline extends to each external git root — record its `git status --short` before/after the same spawns: an added root is an accessible filesystem surface for the spawned role, so a read-only-intent root is verified unchanged, never assumed. The check follows the **spawning session** — when the executor spawns peers in-session (loop step 4), the executor runs `/peer-loop`'s per-round checks itself. Under `isolation=worktree` the check runs the two-role split from preflight step 8: the WORKTREE's status is the edit-surface check (an executor's edits appear there), and the BASE checkout's status is verified separately as the non-interference proof — an unexpected BASE change stops the loop just the same (see **Run isolation (worktree-per-run)**). An unexpected change stops the loop.
- **MUST-PAUSE surfacing from a spawned executor.** A spawned executor cannot block waiting for a human: when a MUST-PAUSE gate fires inside its run (including inside an in-executor peer pass), it records the open gate in the plan's `Current State / Handoff Note` and its final message, then ENDS its run; the composer relays the gate to the user and resumes per the disposition. This is the canonical definition — `/peer-loop`'s orchestrated mode points here.
- **Inconclusive over unsafe.** If the cross-family peer cannot run, the peer pass is inconclusive — never fall back to a same-family peer.
- **Composer-shell failure — probe the degraded surface before pausing.** When the COMPOSER's own shell stops executing commands (distinct from an executor failure — e.g. repeated no-exit-status failures on trivial commands), probe whether subagent shells still work before pausing the run: a working subagent shell is a legitimate degraded surface for composer-owned duties (commits, smokes, spawn wrappers). Guardrails on the degraded surface: the subagent runs EXACT composer-specified commands only — no autonomous edits, planning, or scope discretion — and every existing gate is preserved unchanged (commit authorization, confirmed-grants-only — a SCOPED-tier gate: under `perms=bypass` there is no grant list and the close-table confirm was the entire injection gate, while every OTHER gate named here stands on both tiers — user confirmations, filesystem-escape checks — the composer still verifies the escape baseline from the subagent's reported `git status`). Record the degradation in the run-state note (Visibility + run-state) and continue; pause only when both surfaces are dead (observed live 2026-07-05: the composer session's shell died post-handoff; a shell subagent performed the authorized phase commit and the run continued).
- **Composer duties are never asks.** Three duties are the composer's unconditionally and are never offered to the user as options: (a) monitoring/polling a spawned role — the wake layers named by the preflight-resolved `Wake:` line (Preflight step 10: the Layer-1 output watch, the Layer-2 monitor subagent, or the sleep-sentinel fallback — see the wake contract in Visibility + run-state) are armed at spawn, always, with the `WATCH: armed` artifact printed as proof ("Want me to start polling?" is the anti-pattern; observed live); (b) `/document` — performed per loop step 7 and at phases-done (see Termination), never offered as a menu item; (c) phase-boundary closeout + auto-advance — under `scope=all`, closing out a finished phase and spawning the next phase's executor is performed, never offered ("Say continue for composer closeout + the next spawn" is the anti-pattern; observed live 2026-07-05). This clause removes duty-shirking asks ONLY: MUST-PAUSE gates, the live-user-smoke pause, user-only environment verification legs (e.g. a toolchain the agent's host lacks), usage-exhaustion recovery, and every Termination stop remain legitimate waits.

## Usage-exhaustion recovery (claude pools)

When a `claude -p` role's run actually FAILS on a usage limit — a limit-rejection error mid-run, or a spawn refused on an exhausted pool; detected via the heartbeat's tier-1 error-string / `rate_limit_event` watch or an exit with no `ROLE: handoff` line — the composer MUST-PAUSEs and offers three options. The trigger is the FAILURE, never a probe percentage: the usage checkpoint stays warn-only (see Visibility + run-state), and this section adds recovery from a hard failure, not a threshold action.

```text
The <role>'s claude account pool is exhausted (run failed on a usage limit).
1. Wait — the pool resets at <reset time — the structured session/week resets fields, headline text when unparsed, or resetsAt>; resume after it passes
2. Switch account — continue the same session under a different Anthropic account
3. Continue in composer — the live session takes over as the executor
```

- **Wait** — hold the run; after the reset time passes, resume the same role (`--resume <session-id>` — the roles in this section are `claude -p` by its trigger) and continue where it stopped.
- **Switch account** — pick from the switch menu, reconciled to the Q1R display shape (structured session/week % + reset times alongside each identity, freshness-sorted idle-first per the sharing advisory's recent-repos signal and its completeness rule — only complete snapshots rank as idle; DISPLAY reconciliation only: the usage checkpoint's log-line format in Visibility + run-state is byte-unchanged), or run the wizard's guided new-profile flow (pin-the-binary + bogus-dir probe + browser OAuth + the `.credentials.json` artifact check), then **transfer the session** (live-verified 2026-07-03; recipe detail in `docs/integrations/cross-agent-orchestration.md` → multi-account): copy the session transcript jsonl (its filename IS the session id) plus the session-env entry from the old profile's `projects/` store into the new profile's matching paths, `--resume <session-id>` under the new profile's `CLAUDE_CONFIG_DIR`, and run a short alive-check. Then rewrite the `Loop config:` line (replace-not-stack) to the new profile and write a fresh `ROLE: start` carrying the new `profile=` key — the failed role's `ROLE: end` records the limit failure as its result. If the resume is rejected, fall back to a **fresh executor from the plan handoff note** — no work is lost either way, since all code, findings, and plan state are on disk.
- **Continue in composer** — the live session takes over as the executor for the rest of the phase. Re-validate the cross-family rule first: if the composer's model family no longer differs from the configured peer's, the peer pass is inconclusive — stop for direction (Preflight step 7's rule), never silently proceed same-family.

A `claude -p` **peer** hitting a limit needs no session transfer — peers are fresh per round. Offer the same three options, where "switch account" simply respawns the round fresh under the chosen profile and updates the peer row in `Loop config:`. The stall watchdog's check step routes here too: a role that died on a limit is DEAD-on-limit (recover per this section), not hung — do not kill-and-report it as a hang.

## Auto-commit (git-safety)

Default **ON**; opt-out at the wizard (Q5). The wizard close-table confirm is the commit authorization — see the wizard **Close**. When on, each per-phase commit obeys the repo Git Safety rules:
- stage ONLY that phase's files, by EXACT path — `git add -- <path> [<path> …]` — never `git add .`/`-A`/`-u` or directory/pattern adds: on a tree carrying parallel work (another run's edits, retained dirt, a multi-track working tree) anything broader sweeps foreign files into the phase commit (observed downstream),
- surface `git status` + a diff summary before committing,
- use a clear, human-sounding commit message (no AI / workflow jargon),
- and **never push**.

Under `isolation=worktree`, staging and the commit run in the WORKTREE checkout on the run branch — the worktree plan copy included when the phase changed it (see **Run isolation (worktree-per-run)**) — and the branch-`main` warning below is moot there: commits land on the run branch and reach the base only via the canonical merge-back close path.

If the branch is `main`, warn that a later manual push may deploy before the user pushes.

## Termination

Stop and report when any of these holds:
- **Phases done** — no phases remain; the composer PERFORMS the run-end duties itself, never offers them as menu options: it runs the completion handoff (the plan's completion-state decision pass via `/execute`), runs the final interactive `/document` pass (per loop step 7), and — when `autocommit=on` — makes the final commit. The only run-end action it RECOMMENDS is `/retro` (the run report: what happened, what it cost, what failed and why) — under `isolation=worktree`, pointed at the run's base-side log root `.cursor/loops/<iso-id>/` for logs and at the WORKTREE for the plan (see **Run isolation (worktree-per-run)**). The remaining user calls are push (always the user's explicit call) and any user-only environment verification legs.
- **Stalemate** — a phase's `/review-loop` or `/peer-loop` produced no new verified progress; stop and report rather than burn rounds.
- **Cap reached** — a composed loop hit its max-round cap; report remaining findings and offer to raise the cap for one more round. Residue routing: recommend raising the cap one confirmation round when the final round's unconfirmed residue touches code or shipped prose at MED+, and accepting cap-closure when the residue is records-only plan-file bookkeeping — a release-end whole-diff hardening pass re-covers it (rule proven across three live cap gates, 2026-07-10).
- **MUST-PAUSE** — a gate surfaced for the user; the loop waits for the disposition.
- **Review-integrity failure** — `/review-loop` returned without reporting its per-round `Review History` integrity check (the composer's step-3 assert); stop and surface rather than run blind on skew/escalation.
- **User stop** — the user asked to stop, or the scope was one-phase-and-stop.
- **Isolation close (every stop path above).** Under `isolation=worktree`, every termination resolves through the per-close-path outcome table in **Run isolation (worktree-per-run)**: a green Phases-done close with `merge=auto` runs the canonical merge-back preconditions there; every other stop retains the worktree/branch/journal with the worktree plan authoritative and the handoff stating the continuation. Never a push.

Never loop past a stop condition without explicit user direction.

## Visibility + run-state

- **Per-role visibility.** Tee each spawned role's output to `.cursor/loops/stage-<N>-<role>.log` (gitignored) and keep a concise per-phase status line in the chat (phase, backend, review/peer outcome — including the `Review History` integrity-check report — smoke result, commit state). Do not paste full transcripts into the chat.
- **Progress heartbeat (two-tier).** While a spawned role runs, the composer monitors the plan's task markers plus the role's streaming surface — the tee log for CLI backends; the transcript/progress surface for Cursor subagents. For a `claude -p` role, spawn with `--output-format stream-json --verbose` teed: text mode buffers its narrative to exit (a tee log can sit unchanged for a whole run, leaving nothing to watch), while stream-json emits per-event JSON lines in real time — including a live `rate_limit_event` line (verified 2026-07-03). Initialize the baseline read silently BEFORE the first poll (an empty baseline otherwise fires a fake first transition). Then signal on two tiers — prefixing EVERY composer-posted Tier-1, liveness, and status chat line with `[<runkey>]`: the breadcrumb is the RESUME ANCHOR (a recovering composer greps recent chat for the bracketed runkey to re-find its run — see the Resume contract below), and the runkey is the phase's log prefix and already encodes the phase, so no `p<N>` suffix. **Tier 1 — post a one-line chat update:** a task-marker flip, naming the flipped task (diff the task lines, don't just count markers) with its elapsed time in state ("Task 8 🟨 for 42m" — cheap stall detection for free); a new `ROLE:` / `DELEGATION:` line including `ROLE: round` and the `ROLE: handoff` exit sentinel; a peer round completing; a known limit/error-string match (e.g. "You've hit your session limit", a `rate_limit_event` with a blocked status — route to Usage-exhaustion recovery within seconds instead of after a poll-and-sleep cycle). **Tier 2 — log-only, never notify** (the default): file-count ticks, byte-growth liveness, and other "still working" signals — they are information-free as notifications. One Q5-gated exception: when the wizard's Q5 liveness setting is ON (backend-conditional default — ON for CLI executor backends, OFF for Cursor-subagent / live-session; interval default 10 minutes), a window of N minutes with no Tier-1 anchor posts ONE chat liveness line built from the monitor subagent's snapshot — or from the Tier-2 signals on the sleep-sentinel fallback (see the wake contract below) — e.g. `executor alive — log +12KB, last event 3m ago`, then stays quiet until the next window or anchor. The liveness line is informational only — the stall watchdog below keeps sole ownership of hang classification — and transcripts are still never pasted. No new log prefix — the heartbeat keys on these existing anchors only. **Turn-based orchestrator mechanics — the two-layer wake contract:** an event-driven composer (e.g. a Cursor composer session) never wakes between turns on its own — "monitoring" must be implemented, not assumed. The host's wake modes are resolved ONCE at preflight into the required `Wake:` line (Preflight step 10); every arm below follows THAT resolution, and the artifact grammar + the return-handling duty live in **Wake arming artifacts** below. **Layer 1 — the attached watch (the exit wake).** The CLI role runs as a composer-owned background shell per loop step 1's attached-spawn mandate, with its output watch configured AT SPAWN to fire on the FULL Tier-1 anchor set — any `ROLE:` / `DELEGATION:` line plus the limit/error strings, not just the `ROLE: handoff` exit sentinel (a handoff-only watch leaves entire review/peer passes silent; observed live 2026-07-04) — and the shell's completion notification wakes the composer the moment the role exits, so handoff, crash, and limit-death are detected within seconds. Free and instant, but host-conditional (`layer1=bg-shell`, resolved at preflight — it needs a persistent composer with a background-shell watch surface) and regex-blind. **Meta-repo caveat — quoted anchors are advisory:** in a repo whose own content documents the anchor vocabulary (quoted `ROLE:` / `DELEGATION:` prose in skills or docs — e.g. a repo that maintains the workflow skills themselves, or any run whose roles read or edit the workflow files), tee-log anchor matches from tool-result echoes are EXPECTED false positives: a role reading or editing that prose echoes the quoted lines into its stream (observed live 2026-07-05: ~80 quoted-prose matches saturated the watch in one run). Treat anchor matches as ADVISORY there — key exit classification on the shell's completion notification plus the role's own final output line (the `ROLE: handoff` sentinel it writes as its last act), and have the monitor's snapshot distinguish a genuinely-new trailing anchor from a mid-stream quoted echo — the concrete stream-json discriminator: a REAL event is an unescaped line-start JSON object (`{"type":"rate_limit_event"`), while a quoted echo inside tool-result content carries escaped quotes (`\"type\":\"...\"`); filter EVENT-shaped anchors (the `rate_limit_event` family) on the unescaped line-start form plus, for limit wakes, a non-`allowed` status, since the healthy stream emits routine `"status":"allowed"` rate-limit events — text anchors (`ROLE:`/`DELEGATION:`) live inside message JSON on both real and echoed paths and stay advisory per this caveat (observed live 2026-07-10: two false limit-string wakes, retuned to zero for the rest of the run). **Layer 2 — the monitor subagent (default-on for CLI executor backends on hosts with cheap background subagents).** `layer2=monitor` is the abstraction, not one implementation: it admits the return-per-event monitor SUBAGENT (the canonical shape, fully described here) and a PERSISTENT harness-native watch tool in the same slot — one that emits per-event notifications instead of returning and never self-terminates (its arm-proof and writer clauses are in **Wake arming artifacts**; the exit-branch teardown there is mandatory for it). At spawn the composer also arms a cheap-model background monitor subagent with an explicit prompt contract: the executor PID, the tee-log path, the plan path, the Tier-1 anchor regexes, the liveness window (the Q5 interval), the current watch baseline (the tee-log offset / last-seen anchor plus the plan task-marker snapshot at arm time), the current wall-clock time at arm (the monitor's timestamp anchor for its `MONITOR:` `ts=` values), and the canonical `MONITOR:` snapshot grammar it MUST write (**Wake arming artifacts**). The monitor polls its inputs every ~60–90 seconds and RETURNS on the first NEW Tier-1 anchor (past its baseline), executor exit, liveness-window expiry, or anomaly (`event=anomaly` — exactly two string-shaped triggers plus a judgment catch-all, defined in **Wake arming artifacts**; informational, never a monitor-side kill) — return-per-event, never one long-lived poller (a long-lived monitor accumulates context and wakes the composer only once) — carrying its one-line `MONITOR:` snapshot in the canonical grammar: current task + time-in-state, log growth, last event age, the return event. The prompt contract also includes a **dead-at-spawn startup check**: within the monitor's first one or two polls, verify the spawn actually started — the tee log exists and is growing past the spawner-written `ROLE: start` line, and the PID is alive; if not, RETURN `dead-at-spawn` immediately instead of waiting out the liveness window (observed live 2026-07-06: an `ENOENT`-failed spawn sat undetected for a full 10-minute window; the same check on the respawn confirmed a healthy start in ~75 seconds — a fresh spawn is at its most fragile in its first minute, and the liveness window is tuned for mid-run quiet, not startup failure). The prompt contract's probe-failure rule: when the monitor's OWN probe commands fail — the log read errors, the PID check errors — it RETURNS `event=anomaly` naming its failing probe, never keeps polling silently: a watchdog keyed on a broken probe reports permanent quiet, indistinguishable from a healthy idle run (observed downstream: hours of silent polling on a failing probe command). Each return wakes the composer, which handles it under the ATOMIC, CONDITIONAL return-handling duty in **Wake arming artifacts** — a non-exit return posts the update or liveness line AND re-arms a fresh monitor for the next window with an UPDATED baseline in the same handling turn; an exit return closes the role (`ROLE: end` first, terminal classification, NO re-arm) — a monitor reports only anchors new since its baseline, so a re-arm never re-fires on the anchor that triggered the previous return (preflight rotation guards against prior-run anchors; the baseline guards against this run's already-handled ones); the monitor's `MONITOR:` snapshots append to `.cursor/loops/<runkey>-probe.log` (gitignored; line 1 is the phase's `Wake:` header per Preflight step 10; monitor probes are NOT `ROLE:` activations and write no `ROLE:` lines). Q5 liveness OFF disables the liveness-window-expiry returns and the chat liveness line, not the anchor/exit returns. Layer 2 adds what the watch alone cannot: semantic snapshots, the stall watchdog's observation input, and cover when the native watch fails or a spawn must survive composer restart (where it is REQUIRED — loop step 1). **Sleep-sentinel fallback (`layer2=sentinel` — hosts without background subagents):** schedule a wake timer per quiet window — a backgrounded sleep-N-minutes-then-print-sentinel shell whose output wakes the composer to run the quiet-window check and post the liveness line. Each firing is consumed under the return-handling duty in **Wake arming artifacts**: role live → the next one-shot timer is armed in the SAME handling turn, each arm printing its fresh `WATCH: armed … layer2=sentinel` line; role exited → `ROLE: end` first, no re-arm; arm failure → the `layer2=none reason=` walk. A continuously-running orchestrator may simply poll; either way every liveness line and watchdog observation must have a concrete wake path — a quiet-window rule with no wake mechanism is a no-op. Cost note (informational, no behaviour change): anchor-driven blocking waits and monitor returns cost nothing while waiting, but each wake is one composer turn over the run's session context — the reason wakes key on Tier-1 anchors, monitor returns, and long quiet windows rather than tight polls, and worth knowing when the composer runs on API billing.
- **Wake arming artifacts (3-family grammar — canonical definition).** The wake contract's state is carried by THREE required greppable artifact families — `Wake:` (one authoritative declaration per run, resolved at Preflight step 10; a Resume-contract recovery re-resolves and reprints it, and the NEWEST `Wake:` line supersedes prior ones — consumers read the last, never the first), `WATCH: armed` (one per CLI-role spawn at arm time, PLUS one fresh line whenever the armed Layer-2 mechanism CHANGES thereafter, PLUS one per one-shot sentinel arm — a sentinel is CONSUMED by firing, so every sentinel re-arm is a fresh arm and prints its own line — always carrying the ACTUALLY-armed state; printed by the arming session — loop step 1's arming artifact), and `MONITOR:` (one snapshot line per monitor EVENT, written by the MONITOR ITSELF — the return-per-event subagent writes it as its last act before returning; a persistent watch tool writes it per event notification, no return existing (the two-shape clause on Layer 2); this is the canonical snapshot prefix — two live runs wrote divergent `MONITOR:`/`PROBE:` prefixes for the same snapshot, so one grammar replaces both) — alongside the existing `ROLE:` family. There is NO `rearmed` and NO `degraded` verb: degradation is the MANDATORY layer-state fields on the `armed` line (`layer2=none` REQUIRES `reason="..."`), and re-arm proof is the PRESENCE of the next `MONITOR:` return — monitor-written evidence beats a composer-written claim, because the party that failed to re-arm is the party that would have written the claim — and a PERSISTENT `layer2=monitor` watch tool (the two-shape clause on Layer 2) needs NO re-arm: its single spawn-time `WATCH: armed` line plus its continuing per-event notifications ARE the arm/liveness proof. Grammar (`layer1` enum: `bg-shell|subagent-harness|fg-wait|native` — there is NO `layer1=none` value: `fg-wait` and `native` have no unwatched-continuation case by construction; `layer2` enum: `monitor|sentinel|none`):

  ```text
  Wake: run=<runkey> layer1=<bg-shell|subagent-harness|fg-wait|native> layer2=<monitor|sentinel|none> [interval=<minutes>m] [reason="<required when layer2=none>"]
  WATCH: armed ts=<iso> run=<runkey> role=<executor|architect|delegate|peer|advisor> layer1=<...> layer2=<...> [reason="<required for layer2=none and for any mechanism-change line>"]
  MONITOR: ts=<iso> run=<runkey> role=<...> event=<anchor|liveness|anomaly|exit|dead-at-spawn> task="<current task + time-in-state>" log=<growth> last=<age> detail="<one-liner>"
  ```

  ONE end-to-end worked example (a healthy phase, spawn to closure, merged chronologically across the probe and role logs — the second `MONITOR:` line IS the re-arm proof for the first return's handling; the exit return closes the role: the spawner's `ROLE: end` is its FIRST handling write, and no `WATCH:` line follows it):

  ```text
  Wake: run=lr29-2 layer1=bg-shell layer2=monitor interval=10m
  WATCH: armed ts=2026-07-12T09:02:11+10:00 run=lr29-2 role=executor layer1=bg-shell layer2=monitor
  MONITOR: ts=2026-07-12T09:14:40+10:00 run=lr29-2 role=executor event=anchor task="Task 7 🟨 for 12m" log=+18KB last=40s detail="new ROLE: round reviewer"
  ROLE: handoff ts=2026-07-12T09:25:48+10:00 run=lr29-2 reason=phase-complete
  MONITOR: ts=2026-07-12T09:26:05+10:00 run=lr29-2 role=executor event=exit task="Task 8 🟩" log=+31KB last=5s detail="shell completed; final line ROLE: handoff"
  ROLE: end ts=2026-07-12T09:26:18+10:00 run=lr29-2 role=executor result="phase complete — handoff observed; closed end-first, no re-arm"
  ```

  **Writers:** the `Wake:` line and every `WATCH: armed` line are written by the ARMING session — the composer for every loop-spawned role (chat at preflight; line 1 of each phase's `<runkey>-probe.log` at that phase's first arm — a dry-run writes no file), and an executor arming its own foregrounded peer spawns (loop step 4) — or Mechanism-C consult spawns (Optional advisor) — likewise writes its own, with its own truthful state — `layer1=fg-wait layer2=none reason="foregrounded synchronous wait"` by construction, the obligation riding the loop step 1 spawn-prompt carry (a fresh executor never reads this section unprompted); the monitor writes every `MONITOR:` line — the return-per-event subagent appends it to the probe log as its last act before returning; a persistent watch tool writes the SAME `MONITOR:` snapshot grammar per event notification, no return existing (monitor probes remain non-`ROLE:` activations and still write no `ROLE:` lines). **Anomaly returns (`event=anomaly`) fire on exactly TWO string-shaped triggers** — (1) repeated-identical-error lines, ≥3 in one poll window; (2) a permission-denial storm: repeated `was blocked. For security` tool-result lines (verbatim denial shape observed live 2026-07-10, single sample: `<action> … was blocked. For security, …`) or a growing structured `permission_denials` array in a stream-json role's result events (the deterministic channel, no string match needed) — plus a judgment catch-all: anything else clearly anomalous (e.g. a visible write to `.env`/secrets) — flag it, the composer classifies. Spawn invocations tee stderr into the role log (`2>&1`) so stderr-only warnings reach the monitor's input (observed live 2026-07-10: a CLI capability warning printed to stderr only). Anomaly returns are INFORMATIONAL — composer check-then-classify, never auto-kill; off-plan-surface WRITE detection stays owned by the deterministic git filesystem-escape baseline (Safety gates), never the monitor. **The return-handling duty (ATOMIC and CONDITIONAL — it covers EVERY Layer-2 wake: monitor returns AND sentinel firings).** A Layer-2 wake is not fully processed until, in the SAME handling turn — for a NON-EXIT wake (a monitor's `event=anchor|liveness|anomaly` return, or a sentinel firing that finds the role live): the composer posts the update AND arms the next watch per the resolved `Wake:` mode — for a sentinel that means arming the next one-shot timer AND printing its fresh `WATCH: armed … layer2=sentinel` line (a consumed one-shot's re-arm is a fresh arm, and its armed line IS the proof: no monitor exists to write future `MONITOR:` evidence; unchanged monitor re-arms still print nothing, the next `MONITOR:` return staying their stronger proof); for an EXIT wake (`event=exit|dead-at-spawn`, or a sentinel firing that discovers the role exited): the observing/spawning session (the composer, for loop-spawned roles) writes the role's `ROLE: end` FIRST (the end-first clause on the `ROLE:` bullet below), posts the update, runs the terminal classification (handoff / crash / limit-death per the existing rules), and explicitly does NOT re-arm — the role is CLOSED; never a watch on a dead PID — and, in the same handling turn, CANCELS/CONSUMES every still-armed wake mechanism for that role that did not self-terminate with the exit return: an armed-but-not-yet-returned monitor (when Layer 1 reported the exit first), a pending sentinel timer, and a persistent harness watch tool (which never self-terminates — teardown is mandatory for it) alike, because a wake mechanism outliving its closed role emits false events against the frozen log (observed live 2026-07-10: two post-exit false stalls). A live-role re-arm failure walks monitor → sentinel → none, and EVERY mechanism change prints a fresh `WATCH: armed` line carrying the actual new state with its transition reason — a successful sentinel fallback prints `WATCH: armed … layer2=sentinel reason="monitor arm failed"`, and only the terminal failure prints the chat-posted `WATCH: armed … layer2=none reason="..."` (quiet-hang detection OFF, stated in the reason); an unchanged, successfully re-armed monitor prints no fresh line — its next `MONITOR:` return is the stronger, monitor-written proof. The `layer2=none` Layer-1-only continuation is legal ONLY while a Layer-1 watch surface actually exists (`layer1=bg-shell|subagent-harness`; Layer 1 is host-conditional, never assumed). A live role with NO Layer-2 mechanism and NO Layer-1 surface has no concrete future wake: the composer foreground-waits it or MUST-PAUSEs — it never continues dark. The handling turn may not end without one of: a fresh armed watch (a persistent watch still armed satisfies this — the two-shape clause), the explicit `layer2=none reason=` line (Layer-1 surface present), or the closed-role / fg-wait / MUST-PAUSE outcome. (Composer death mid-handling is the resume path's case — see the **Resume contract** bullet below — not this rule's.)
- **Stall watchdog (check-then-classify).** If a spawned role's streaming surface stops growing for ~10 minutes, the composer CHECKS before acting — never kill-on-timeout. (A turn-based composer gets the ~10-minute observation from the two-layer wake contract above: the monitor subagent's liveness-window return IS the observation trigger, its snapshot supplying the streaming-surface evidence, with the sleep-sentinel wake timer as the fallback on hosts without subagents. With Q5 liveness OFF the quiet-window wake is disabled too, so a turn-based composer's autonomous stall detection narrows to anchor/exit wakes — the watchdog's ~10-minute observation applies where a quiet-window wake path is configured; without any concrete wake path, a stalled role goes unnoticed until the user asks.) An alive role that recorded an open gate is PAUSED, not hung: a MUST-PAUSE surfacing looks identical to a stall on the streaming surface, and killing it would destroy a valid wait. The check is backend-shaped — CLI backends: check the pid is alive; Cursor subagent: check the transcript/agent status and cancel/respawn if dead. Kill only a demonstrably-hung process (e.g. a codex stdin-hang), and only by its RECORDED identity — the spawn's own PID (a pidfile or the composer-recorded value) or an exact full-path command match, NEVER a broad `pgrep -f` pattern kill (a pattern sweep matches unrelated processes — another run's roles, the composer's own tooling; observed downstream) — then report what was killed and why. A role that died on a usage limit is DEAD-on-limit, not hung — route it to **Usage-exhaustion recovery** instead of the kill path.
- **Usage checkpoint (warn-only).** At preflight (baseline) and in loop step 9 before spawning the next phase's executor, probe the account pools of the CLI backends in use and report one line per pool, warning when a pool is ≥80% used — the fixed default, overridable per invocation (persisted as the optional `usage_warn=` run-settings key below). The between-phase cadence cannot see mid-phase burn — a single long phase can exhaust a pool that was comfortably under threshold at its start (observed live: 45% at preflight, dead mid-phase) — so additionally: at preflight, warn when a pool's remaining headroom looks smaller than a rough per-phase burn estimate (from the plan's phase sizes and any prior run's observed burn; still warn-only); re-probe when resuming an executor mid-phase; and rely on the heartbeat's tier-1 error-string / `rate_limit_event` watch to catch an in-flight limit death immediately (routing to Usage-exhaustion recovery). The loop NEVER pauses, stops, or throttles on usage numbers — the numbers inform the user; the user decides. Probes (short timeout; any failure = one "probe failed" / "not exposed" line, then continue): **claude** — `"<pinned claude>" -p "/usage" --output-format text </dev/null` (per-profile: prefix with the profile's `CLAUDE_CONFIG_DIR`; the headline `Current session` / `Current week` percentages are official server-side numbers); **codex** — the `app-server` JSON-RPC `account/rateLimits/read` method (pipeline recipe in `docs/integrations/cross-agent-orchestration.md` → Usage / rate-limit probes); **Cursor subagent** backend — "not exposed" (no usage surface exists).
- **`ROLE:` log lines (canonical definition).** Every role activation gets one `ROLE: start` and one `ROLE: end` line in the run's loop logs — the same greppable key=value family as `DELEGATION:` lines, strings quoted:

  ```text
  ROLE: start ts=<iso> run=<runkey> phase="..." role=<composer|executor|architect|delegate|peer|advisor> backend=<cursor-subagent|codex|claude-p|live-session> model="<slug>"
  ROLE: end ts=<iso> run=<runkey> role=<...> result="<one-liner>"
  ```

  `run=<runkey>` is the run's log prefix (here `stage-<N>`). A `start` line MAY additionally carry `effort="<level>"` and/or `profile="<name>"` — additive optional keys, present only when the wizard resolved an explicit effort level or a `claude -p` account profile for that role; absence is legal (inherit-effort / default-profile roles omit them), and `/retro` needs no change — it defers to this definition. The `end` line MAY carry best-effort usage fields — `turns≈<n>` / `tokens≈<n>` / `duration=<seconds or h:mm:ss>` — only where the backend exposes them; absence is legal and is reported as "not exposed", never guessed (Cursor subagents expose no token counts to the composer; `codex exec` / `claude -p` sometimes print usage; timestamps always work). **Writer responsibility:** the spawning session writes the pair for each role it spawns — appended to that role's tee log at spawn and on observed completion (the spawner knows the backend, model, and timestamps), and on a role's exit wake the OBSERVING/SPAWNING session writes that role's `ROLE: end` as its FIRST act of return handling, before the update post and the terminal classification (the exit branch of the return-handling duty in **Wake arming artifacts**); a live-session role (executor/architect) appends its own lines to `.cursor/loops/stage-<N>-executor.log` directly, creating the file if absent. **Timestamp authority:** the composer (the spawning session) is the SOLE timestamp authority for the lines it writes — `start`/`end` `ts=` values carry the spawner's wall clock and are the ordering truth for the run. A spawned role's self-written `ts=` is ADVISORY only: headless sessions emit junk wall-clock (observed live: placeholder `ts=<date>T00:00:00` from two executors in one run), so the spawn prompt passes the current wall-clock time at spawn as the role's anchor (loop step 1), and consumers treat a role-written `ts=` as approximate, never as ordering truth. In-session `/review-loop` rounds are not activations — they log a single per-round `ROLE: round` line (`role=reviewer`) instead (see `/review-loop`). The per-phase status line mirrors each role's backend/model from these lines.
- **`ROLE: handoff` exit sentinel (same `ROLE:` family — no new prefix).** A spawned executor's REQUIRED final output line classifies its exit machine-readably:

  ```text
  ROLE: handoff ts=<iso> run=<runkey> reason=<phase-complete|composer-run|must-pause> [gate="<one-liner>"] [suites="<what the composer should run>"] [consults=<reserved>/<cap> refusals=<n> dangling=<n>]
  ```

  Written by the role itself as its last act (unlike `start`/`end`, which the spawner writes) — its `ts=` is therefore advisory per the timestamp-authority rule above: anchored on the spawn prompt's wall-clock carry plus elapsed time, useful but never ordering truth. The optional consult counters appear only on advised executors (Optional advisor); the composer reconciles them against the phase's `CONSULT:` ledger. `reason=phase-complete` = the phase's work is done; `reason=composer-run` = the executor needs the composer to run named commands and resume it (see loop step 3); `reason=must-pause` = an open gate recorded per the record-and-end contract. **Absence of a handoff line on exit = a crash or limit-death** — the composer classifies via the heartbeat's error-string watch and routes limit-deaths to Usage-exhaustion recovery; without it, exit classification is manual archaeology (observed live: a failed resume cost ~7 minutes before detection).
- **Run-state persistence.** After each meaningful step, update the plan's `Current State / Handoff Note` with the current phase, the last completed step, the immediate next action, open blockers, and the last sync — so the composer survives a compaction or restart and resumes from disk, not memory (the full recovery sequence is the **Resume contract** bullet below). The next-action line must reflect the ACTUAL state at write time — never leave a stale "on resume: `/review-loop` → `/peer-loop`" standing after those have partially run (observed live: only the log had the truth, and a composer trusting the plan would have re-run a converged review). Under `isolation=worktree`, these run-state writes — this handoff-note update and the `Loop config:` line below — target the AUTHORITATIVE worktree plan copy from creation onward, never the retained base copy (see **Run isolation (worktree-per-run)**). `/execute-loop` adds no new plan section; it reuses the existing handoff note.
- **Resume contract (composer context loss ≠ run over — canonical recovery sequence).** Trigger: a compaction or session restart — OR the composer simply cannot state the current runkey, phase, and immediate next action (the inability IS the detector; do not wait for a harness signal). Recover in this order, from disk truth, never remembered intent: (0) registry discovery — from the BASE checkout, check `.cursor/loops/*-isolation` journals (and `*-active` markers) for this run's record; a journal holding worktree authority redirects every later step's plan reads AND writes to the recorded authoritative worktree plan copy, and the journal-recorded log root is where this sequence's logs live; reconcile the journal against `git worktree list`, refs, the recorded base state, and any extant close lock (the close-lock recovery rule) BEFORE any merge, authority switch, or deletion — stale/ambiguous ⇒ retain all artifacts, stop, and surface (canonical: **Run isolation (worktree-per-run)**); a journal-less `*-active` marker claiming this checkout is NOT skipped — journal absence is `branch`/`none`'s steady state, while a journal-less `mode=worktree` marker marks a PRE-JOURNAL crash (the claim-to-journal window; total disposition in the ownership-marker bullet, **Run isolation (worktree-per-run)**): validate its run-lifetime owner and adopt/reconcile it through the serialized ownership protocol BEFORE any edit resumes — a live or unknown-liveness foreign marker warns/refuses (the conservative rule), a conclusively-dead owner's marker gets the ownership-checked transition (the worktree-crash case only with ZERO matching git/isolation artifacts — any artifact retains + surfaces), and no journal/worktree/close-lock lifecycle is introduced for the `branch`/`none` modes; NO applicable journal AND NO applicable marker = no isolation machinery, continue; (1) re-read this skill's RUN-MECHANICS sections — the per-phase loop, Safety gates, and Visibility + run-state, PLUS the **Optional advisor** section when the parsed `Loop config:` carries an `advisor=` group — NOT the wizard, whose result is already persisted in `Loop config:`; a whole-file re-read only if the context budget clearly allows it (a full re-read of this skill is large enough to re-trigger the very compaction being recovered from); (2) re-read the plan's `Current State / Handoff Note` and parse the `Loop config:` line — the wizard's persisted resolution — and, when an advisor is configured, parse the current phase's `CONSULT:` reserve/outcome/refused ledger (Optional advisor): recompute the remaining consult budget from the reservations, and CLASSIFY any dangling reservation BEFORE any new consult or executor resume — an in-flight advice return is re-delivered over the pinned per-backend return route, never blindly re-invoked or re-billed; (3) re-run wake-mode resolution (Preflight step 10) and reprint the `Wake:` line — the reprint SUPERSEDES the preflight declaration (newest wins, per **Wake arming artifacts**) — then re-arm the current role's watch per the return-handling duty, checking the dark window first — read the tail of the role's tee log and the probe log for anchors that fired while no watch was armed; an EXITED role gets its `ROLE: end` written FIRST (the exit branch in **Wake arming artifacts** — never re-arm a closed role); (4) resume from what the plan and logs prove, posting a `[<runkey>]`-prefixed status line stating the recovered phase + next action. The `[<runkey>]` breadcrumbs on prior Tier-1 lines are the chat-side anchor back into the run (the heartbeat's breadcrumb obligation above).
- **`Loop config:` line (wizard-config persistence).** The wizard's resolved configuration persists in the same handoff note as ONE `Loop config:` line, written when a real run starts (the wizard's Persistence rule; never during dry-run). Replace-not-stack: overwrite any existing line, never add a second; treat an unparseable line as absent. Forward-compat: parsers read the key-groups they RECOGNIZE and ignore unknown ones — only a line with NO parseable groups is treated as absent, so an older skill meeting a newer line (e.g. one carrying `perms=`/`advisor=`) keeps the groups it understands rather than discarding the whole config. Grammar — semicolon-separated `key=value` groups in the `ROLE:` vocabulary. Per-role group: `<role>=<backend> model="<slug>" [effort=<level>] [profile="<identity>" profile_dir="<dir>"]` — `profile_dir` stores the invocable `CLAUDE_CONFIG_DIR` path (the identity alone is display-only and cannot be re-invoked); the default account stores `profile=default` with no dir. The OPTIONAL `advisor=` group is a role group like any other (`advisor=<backend> model="<slug>" [effort=<level>] [profile="<identity>" profile_dir="<dir>"]`) naming WHO advises the executor/architect — the mechanism (N vs C) is RE-DERIVED from the advised backend + the advisor row at spawn time, never stored (see Optional advisor); the optional `effort=`/`profile=` keys are Mechanism-C-only — a native-N advisor stores backend + model only, since its effort and billing are the advised session's (the second worked example's shape). Run-settings keys: `architect=on|off` (when `on`, the executor group IS the architect and a `delegate=` group is present — the delegate inherits the architect's effort + account, so it carries no effort/profile keys), `cadence=`, `caps=review:N,peer:N`, `scope=`, `autocommit=`, `isolation=worktree|branch|none` + `merge=auto|off` — the Q5 run-isolation pair, both persisted once resolved (absence = the item is ASKED on the reuse path — a pre-isolation stored line has no choice to reuse — never silently resolved); `merge=auto` is legal ONLY with `isolation=worktree` AND `autocommit=on` (any other combination renders the merge item INAPPLICABLE at Q5, resolved `merge=off`); stored `isolation=`/`merge=` follow their own reuse-path transition rule, parallel to the `perms=` transition table: REVALIDATE before reuse — the stored run's isolation registry (the gitignored base-side run record under `.cursor/loops/`) against the current base checkout (recorded base branch/HEAD vs current; registry state) and key compatibility (a stored `merge=auto` against the stored `autocommit` and `isolation`) — and RE-ASK the item on any mismatch, a stale or ambiguous registry, or an incompatible stored combination; NEVER silently reuse an isolation choice — the OPTIONAL `perms=scoped|bypass` — BACKEND-CONDITIONAL: persisted only when the executor/architect backend has a supported bypass rendering (`claude -p`, `codex`); absence = scoped; a stored value follows Q5's pinned transition table on every wizard path (a stored `perms=bypass` whose executor backend resolves to a no-rendering backend is STALE and is normalized out) — the OPTIONAL `usage_warn=<percent>` — written only when the user overrode the usage checkpoint's default warn threshold (80), so a composer restart keeps the override; absence = default 80 — the OPTIONAL `liveness=<minutes>` — a positive integer, written only when the Q5 liveness setting is ON; absence = off; treat a nonpositive or unparseable stored value as stale and re-ask that item on the reuse-last path (the stored key's semantics are unchanged by the backend-conditional Q5 default — only the wizard's pre-fill varies by executor backend) — and the OPTIONAL `verify=composer` — PREFLIGHT-written, never wizard-asked: recorded when the preflight harness probe fails (Preflight step 6), it marks the run as composer-run verification up front; absence = the executor runs its own granted verification commands. It is re-derived by the probe each run — a fresh probe result supersedes any stale stored value, so it is never revalidated on the reuse-last path. Examples:

  ```text
  Loop config: executor=codex model="gpt-5.5" effort=xhigh; peer=cursor-subagent model="claude-fable-5-thinking-max"; architect=off; cadence=every-phase; caps=review:3,peer:3; scope=all; autocommit=on; isolation=worktree; merge=off; perms=bypass
  Loop config: executor=claude-p model="opus" profile="kount" profile_dir="~/.claude-work"; delegate=claude-p model="sonnet"; advisor=claude-p model="fable"; peer=codex model="gpt-5.5"; architect=on; cadence=every-phase; caps=review:3,peer:3; scope=all; autocommit=off; isolation=worktree; merge=off
  ```
